## ① 機能概要

ログインは、利用者が **メールアドレスに届くマジックリンク** で CivilizationOS に入る入口である。パスワードやソーシャルログインは使わず、メール入力 → リンク送信（または開発環境では画面内トークン）→ トークン検証 → **JWT 取得** という一本の経路に固定する。

認証成功後、クライアントは JWT を `localStorage`（キー `civilizationos_jwt`）に保持し、以降の API 呼び出しで `Authorization: Bearer` を付与する。初回利用者は JWT 取得後も **オンボーディング未完了** のままとなり、プロフィール・規約同意を済ませるまで保護ルート内の専用フローに留まる。

本機能は **civilization-os リポジトリ内で完結** する。別リポジトリ IT Hercules Laboratory（IHL）の認証体系とは混在しない。

## ② ユーザーができること

| # | 操作 | 条件・結果 |
|---|------|------------|
| U-1 | メールアドレスを入力し、利用規約に同意して **ログインリンクを送信** する | `POST /api/auth/magiclink`。SMTP 設定時はメール送信。未設定・失敗時は開発用トークンが画面に出る |
| U-2 | メール内リンク（`/login?token=…`）を開いて **自動ログイン** する | フロントが URL トークンを検出し `POST /api/auth/verify` を実行 |
| U-3 | 開発・CI 環境で **「このトークンで認証する」** ボタンからログインする | API 応答の `dev_token` を利用（SMTP 未設定・送信失敗・CI 時） |
| U-4 | ログイン済みの状態を確認し、**アプリへ進む** または **ログアウト（別メールで再ログイン）** する | `/login` に JWT 保持中でアクセスした場合 |
| U-5 | ログイン画面から **初回起動の手引き・全機能一覧** ヘルプへ移動する | 未認証でも閲覧可能なヘルプ導線 |
| U-6 | セッション再開 | 保存済み JWT で `GET /api/auth/me` 等を呼び、有効期限内（既定 **7 日**）は再ログイン不要 |
| U-7 | （初回のみ・ログイン直後）オンボーディングへ進む | JWT 取得後 `onboardingComplete === false` の間、`ProtectedApp` がオンボーディング専用ルートに誘導（詳細は **03 新規登録** 要件参照） |

## ④ 機能要件（番号付き）

### FR-LOGIN-01 メール入力と規約同意

- ログイン画面は **メールアドレス入力** と **利用規約同意チェック** を必須とする。
- 同意未チェックまたはメール未入力時は送信ボタンを無効化する。
- Enter キーで送信可能（同意済み・メール入力済みの場合）。

**実装**: `frontend/src/auth/LoginPage.tsx`  
**採用**: REQ-002

### FR-LOGIN-02 マジックリンク発行

- `POST /api/auth/magiclink` は body に `email` を要求する。
- メール形式不正時は `400 INVALID_EMAIL`、未入力時は `400 EMAIL_REQUIRED`。
- 成功時は `ok: true`、`isNewUser`、`email_sent`、`smtp_configured` を返す。
- SMTP 設定済みかつ送信成功時、リンク URL は `{PUBLIC_APP_URL}/login?token=…` 形式でメールに含める。
- マジックリンクトークン TTL は **15 分**。

**正本**: `design/phases/Phase_auth_login.md` §マジックリンクの寿命  
**実装**: `backend/src/logic/authLogic.ts`（`MAGIC_LINK_EXPIRES_MS`）、`backend/src/api/routes/auth.ts`

### FR-LOGIN-03 開発・CI フォールバック

- SMTP 未設定または送信失敗時、API はログにトークンを出力し、条件に応じて JSON に `dev_token` を含める。
- フロントは `dev_token` がある場合、「このトークンで認証する（開発用）」ボタンを表示する。
- CI（`CI=true`）または `MAGICLINK_E2E_INCLUDE_DEV_TOKEN=1` 時は SMTP 成功でも `dev_token` を含めてよい（E2E 用）。

**実装**: `auth.ts` L105–116、`LoginPage.tsx` §sent ステップ

### FR-LOGIN-04 トークン検証と JWT 発行

- `POST /api/auth/verify` は body または query の `token` でワンタイムトークンを消費する。
- 成功時 `jwt`、`user`（公開形）、`isNewUser` を返す。
- 失敗時は `TOKEN_REQUIRED` / `INVALID_TOKEN` / `TOKEN_ALREADY_USED` / `TOKEN_EXPIRED`（いずれも 400）。
- 検証済みトークンは R2 上で `used: true` に更新し再利用を防ぐ。

**正本**: `docs/auth-login-surface.md` §verify

### FR-LOGIN-05 URL トークン自動検証

- `/login?token=…` で開いた場合、フロントは自動で verify を実行する。
- 成功時はクエリを消去し `/` へ replace 遷移する。
- 失敗時はバナーエラー表示のうえメール入力ステップに戻す。

**実装**: `LoginPage.tsx` L40–63

### FR-LOGIN-06 セッション保持と再開

- JWT は `localStorage` キー `civilizationos_jwt` に保存する。
- アプリ起動時 `AuthContext` が JWT を読み、`GET /api/auth/me` でユーザー状態を復元する。
- JWT 有効期限は既定 **7 日**（`JWT_EXPIRES` in `authLogic.ts`）。

**実装**: `frontend/src/auth/AuthContext.tsx`

### FR-LOGIN-07 未認証アクセスの保護

- 認証必須ルートは `ProtectedApp`（`AppRoutes.tsx`）でガードし、未ログイン時 `/login` へ誘導する。
- オンボーディング未完了時はオンボーディング専用ルートに閉じる（`/help/builder-capability` のみ例外）。

**正本**: `Phase_auth_login.md` §初回オンボーディング

### FR-LOGIN-08 Bearer ミドルウェア

- 保護 API は `Authorization: Bearer <jwt>` を要求する。
- 無 Bearer / 形式不正 → `401 UNAUTHORIZED`
- JWT 無効・期限切れ・改ざん → `401 INVALID_TOKEN`
- ユーザー不在 → `401 USER_NOT_FOUND`（`requireAuthWithUser*` 系）
- カルマ停止 → `403 KARMA_SUSPENDED`（`requireAuthActive` 等）

**正本**: `docs/auth-login-surface.md` §JWT と Bearer

### FR-LOGIN-09 レート制限

- `POST /api/auth/magiclink`: IP あたり **20 回 / 60 秒** 超過で `429 RATE_LIMIT`
- `POST /api/auth/verify`: IP あたり **60 回 / 60 秒** 超過で `429 RATE_LIMIT`
- 複数台構成ではインメモリ限界あり（`docs/auth-threat-rate-limit.md` 参照）。

**実装**: `auth.ts` L30–43、`authRoutesRateLimit.test.ts`

### FR-LOGIN-10 エラー表示（UI）

- クライアントは API の `error` コードを優先し、`humanizeApiCode` でユーザー向け文言に変換する。
- フィールドエラー・バナーエラーに `role="alert"` と `data-testid` を付与する。

**実装**: `LoginPage.tsx`、`frontend/src/lib/apiClientError.ts`

### FR-LOGIN-11 ログイン済み UI

- JWT 保持中に `/login` へ来た利用者にはメール表示・「アプリへ進む」・「ログアウト（別のメールでログイン）」を示す。
- オンボーディング未完了時はその旨を表示する。

## ⑥ MiniKernel / C-USB 上の位置づけ

```
World → FeatureNode: auth → Kernel: login (KernelType.view)
         ↓
     Component: LoginPage / AuthContext（V レイヤー）
         ↓
     Connector (X): POST /api/auth/magiclink|verify, GET /api/auth/me
         ↓
     Kernel (K): authLogic（マジックリンク・JWT・R2 ユーザー読込）
```

| 項目 | 値 |
|------|-----|
| **FeatureNode** | `auth`（FN-01 認証・ユーザー管理） |
| **Kernel** | `login` / `KernelType.view`（レガシー A0） |
| **ルート** | フロント `/login`（入口 UI）。MiniScreen ルーティング表では `/` も `auth.login` に紐づくが、実装のログイン画面正本は `/login` |
| **ITO** | IN: email / magic token → Transform: verifyMagicLink, issue JWT → OUT: jwt + public user |
| **R2 契約** | `auth/email_index/{hash}.json`、`auth/tokens/{token}.json`、`users/{userId}.json`（JWT 自体は R2 非保存） |
| **C-USB** | 認証は Connector + Kernel 層。画面単位設計ではなく Kernel UUID / FeatureNode 経由で fork 可能 |

**参照**: `kernel/src/featureRegistry.ts` FN-01、`frontend/src/kernel/registry.ts`、`指示/2026.3.27/統合仕様書_MiniScreenKernel_v5.md` §FN-01
