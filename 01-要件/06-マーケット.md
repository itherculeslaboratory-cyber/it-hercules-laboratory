## 1. 機能概要

マーケットは、RAG chunk（`chunk_id`）を軸にした **出品・オークション・抽選・プラチナコイン順・テンプレ配布** のチャネル群と、**Q&A・称賛・未出品オファー・ラブレター募集** などの社会機能（engagement）を提供する。取引方式の詳細は [`06-マーケット-取引方式-v1.md`](../02-設計/features/06-マーケット/sub/取引方式-v1.md)。文明 OS は **エスクロー・決済代行を行わない** P2P マッチング基盤であり、取引の透明性は **公開 trade-events ログ** と **カルマ連動** で担保する（`market_governance.csv` `mkg_p2p_core`）。GMO あおぞら連携（REQ-007）は **8% 運営プール積立** の実入金突合用。住所は OS 内に保持せず、局留め等は外部 API ヒントで支援する。

## 2. ユーザーができること

- **出品ライフサイクル**: chunk を固定価格・オークション・オファー募集・**抽選**・**プラチナコイン順**のいずれかで出品し、成約・取下げ・オファー審査を状態遷移で管理する（管理者権限が必要な遷移あり）。
- **オークション**: 締切付き入札・自動決着（`settleDueAuctions`）・入札通知を利用する。
- **抽選出品（TX-LOTTERY）**: 締切まで応募 · 均等乱数で当選者 1 名確定（[`06-マーケット-取引方式-v1.md`](../02-設計/features/06-マーケット/sub/取引方式-v1.md) §3.6 · 出典: アイディア §1 Phase2 · 009 スレッド）。
- **プラチナコイン順（TX-PLATINUM-PRIORITY）**: 定員超過時 **累計 Coin 降順**で割当（Pay To Win 禁止 · アイディア §5 · PlatinumCoinRules §17 · 同上 doc §3.7）。
- **テンプレ市場**: 公開テンプレの閲覧・管理者による INSERT 配布（`fork/v2` 連携は別ドキュメント）。
- **Engagement**（`/market/social`）: 公開 Q&A、称賛、未出品個体へのオファー、ラブレター一括募集の作成・応募・選定。
- **取引ログ**: 取引イベントの参照・記録（`fee_unpaid` 時はカルマ減点・公開ログ）。
- **GMO 連携**（経済マスター）: 期待入金登録・Webhook 受信・照合メタ参照（本番鍵は人間ゲート）。
- **配送支援**: 局留め・郵便番号検索の外部公式導線（`post-office-hold-hint`）。

## 4. 機能要件

| ID | 要件 |
|----|------|
| FR-MKT-01 | **三チャネル**を `marketListingStateStore` / `marketAuctionStore` / `templateMarketStore` で分離管理する（`Phase_market_channels.md` 表）。 |
| FR-MKT-02 | **Listing 状態機械**: `unlisted`→`listed_*`→`sold`/`delisted`/`offer_*` の許可辺のみ `POST /api/market/listing-state/transition` で遷移。不正遷移は 409。 |
| FR-MKT-03 | **現在状態**は `chunk_id` のイベント列末尾 `to_state`。イベント無しは `unlisted`。 |
| FR-MKT-04 | **オークション**: `open` 中のみ入札。`ends_at` 経過で `settleDueAuctions` が `settled`。最高入札者・落札額を確定（入札なしも決着）。 |
| FR-MKT-05 | **Engagement**: `subject_key` 単位で Q&A（質問即公開・回答済みフラグ）、称賛、未出品オファー、ラブレター募集 API（ProjectRules §6.6）。 |
| FR-MKT-06 | **非エスクロー**: 資金預かり・配送保証は行わない（`mkg_p2p_core`）。 |
| FR-MKT-07 | **8% 積立**: **取引成立**（§11.0.1 — 配送完了確認 + 評価確定）後の運営プール義務（売り手振込）。**マッチング確定だけでは発生しない**。未払いは `fee_unpaid` trade-event + **月次 Fibonacci Δcount**（§11.7 · §08 §5.1）。legacy CSV `fee_unpaid_karma_delta` は **IHL 再定義まで参照のみ**。 |
| FR-MKT-13 | **取引成立判定**: マッチング／オークション `settled` **単独では成立しない**。**配送完了確認**（買い手受取申告）**かつ** **評価確定**（手動 or 配送完了から **1 ヶ月** 無評価で自動 **「評価:良い」**）。成立時点で 8% 義務・`trade-events` 確定イベントを発火（§11.0.1）。 |
| FR-MKT-14 | **運による抽選**（TX-LOTTERY）: `listed_lottery` → `lottery.open`（応募受付）→ `ends_at` 経過後 **均等乱数**（CSPRNG）で **1 名当選** → `listing.sold` → Trade Stage 1。認証ユーザー **1 応募/1 listing**（重複 409）。**Coin/PT で当選率を上げない**。落選者へ通知 · 外れ券・回数積み上げ禁止。`allocation_completed` 相当を trade-events + listing-state に append。詳細: [`06-マーケット-取引方式-v1.md`](../02-設計/features/06-マーケット/sub/取引方式-v1.md) **§3.6** · 出典 §1.1。 |
| FR-MKT-15 | **プラチナコイン順**（TX-PLATINUM-PRIORITY）: `listed_priority` → `priority.open`（申込受付 · 定員既定=1）→ `closes_at` 経過後 **`Coin Summary.total_coin_count` 降順**で上位 K 名確定 → `listing.sold` → Trade Stage 1。**PT 消費・Coin 購入による順位操作禁止**（Pay To Win 禁止 · アイディア §5）。同点は `created_at` 昇順（先着）— **人間レビュー待ち**（取引方式-v1 §8）。申込時は **自分の累計 Coin と順位帯のみ**表示可。確定時に順位表スナップショット ID を trade-events に記録。詳細: 同 **§3.7** · 出典 §1.1 · `PlatinumCoinRules` §17.2。 |
| FR-MKT-08 | **住所非保持**: 主体識別は UUID / UserID / ユーザー名。配送は匿名配送・局留め推奨（`mkg_address_privacy`）。 |

| FR-MKT-11 | **公開取引ログ**: `GET/POST /api/market/trade-events` — 透明性・監査用 append-only。 |
| FR-MKT-12 | **UI**: `/market/*` ルート群（browse タブ: 出品 · オークション · **抽選** · **優先順** · テンプレ — 草案）· `/market/social`（engagement 権限表は `docs/market-engagement.md`）。TX-LOTTERY / TX-PLATINUM-PRIORITY の遷移は [`06-マーケット-取引方式-v1.md`](../02-設計/features/06-マーケット/sub/取引方式-v1.md) §5 UI 面マトリクス。 |

### 11.0 取引ステージモデル（ユーザー確定 · 2026-06-07）

> **核心**: マッチング前は **公開**（出品・入札・Engagement Q&A）。マッチング後は **当事者二人だけのプライベートボード** で P2P 完結 — **原始メルカリ** 型。**取引成立**（§11.0.1）はその後段 — 配送完了確認と評価で確定する。

| 段階 | 名称 | 内容 |
|------|------|------|
| **Stage 0** | マッチング前 | 出品・オークション・テンプレ · 公開 Q&A（Y09 の指摘パターン 1） |
| **Stage 1** | マッチング後 | **プライベートボード**（第三者非公開）— 振込・発送調整。OS は非エスクロー（§1 · `mkg_p2p_core`）。**成約・`settled` はここで発生 — 取引成立ではない** |
| **Stage 2** | 配送完了 | 買い手による **受取確認**（「荷物届きました」等）· **評価ウィンドウ** 開始（最大 1 ヶ月） |
| **Stage 3** | 評価確定 → **取引成立** | 手動評価 **または** 1 ヶ月無評価での自動 **「評価:良い」**（§11.0.1）。**8% 貢献費クロック起点** |

| ルール | 確定内容 |
|--------|----------|
| **宅配便匿名配送** | **現時点では採用しない**（規模後に将来検討）。`mkg_address_privacy` の住所非保持・局留めヒントとは別軸 |
| **双方合意キャンセル** | 「やっぱやめましょう」双方 OK → **記録なし**（trade-event / 争い / カルマなし） |
| **支払期限** | マッチング後 **2 週間**。振込未確認なら **売り手がクローズ可** |
| **期限切れ買い手** | **Δcount +1**（§08 · §11 §6.4）· **タグ付き悪評価** · **マイページで他ユーザー閲覧可** |
| **配送完了後の争い** | 話し合い · 任意 **公開エクスチェンジ** · プラチナ投票（Y03/Y06）— **常に記録**（判例 R2） |

#### 11.0.1 取引成立判定（ユーザー確定 · 2026-06-07）

| 項目 | ルール |
|------|--------|
| **定義** | **取引成立** = 配送完了確認 **かつ** 評価確定。**マッチング確定・オークション `settled`・listing `sold` 単独では成立しない** |
| **配送完了確認** | 買い手が **受取申告**（UI 例: 「荷物届きました」）— Stage 2 開始 |
| **評価確定** | (a) 当事者の **手動評価**（良い／普通／悪い等 · 詳細設計 TBD） **または** (b) **自動評価** |
| **自動評価** | 配送完了確認から **1 ヶ月以内** に評価がなければ、システムが **「評価:良い」** を自動設定 |
| **判断・理由** | **1 ヶ月無言 = 問題なかった** とみなす。**無期限 pending を防ぐ**（評価待ちのまま 8% や争い境界が曖昧になるのを回避） |
| **8% 貢献費** | **取引成立時点** から義務発生 · 振込案内表示 · `fee_unpaid` 月次起算（§11.7） |
| **用語の区別** | **成約／マッチング確定**（Stage 1）≠ **取引成立**（Stage 3）。Y09「**支払後**」は **代金振込後**（Stage 1 内）を指し、取引成立とは **別タイミング** |

**タイムライン（確定メモ）**

```
マッチング（Stage 0→1）
  → プライベートボード · 振込（2 週間期限 · Y01/Y02）
  → 発送 · 受取確認（Stage 2）
  → 評価ウィンドウ（最大 1 ヶ月）
  → 手動評価 or 自動「良い」
  → 取引成立（Stage 3）→ 8% 貢献費クロック開始
```

#### 11.7.2 未払いとカルマ（月次 Fibonacci Δcount）

未払い状態が **暦月単位** で継続するたび、**その月の Δcount** を Fibonacci 列で付与する。**他のカルマイベントと重複・加算**（排他しない）。

**当月 Δcount 表**（未払い継続月 n に対する **その月の count 増分**）:

| 未払い月 n | 当月 Δcount | Fib(n) |
|------------|-------------|--------|
| 1 | **+1** | 1 |
| 2 | **+1** | 1 |
| 3 | **+2** | 2 |
| 4 | **+3** | 3 |
| 5 | **+5** | 5 |
| 6 | **+8** | 8 |
| 7 | **+13** | 13 |
| 8 | **+21** | 21 |
| 9 | **+34** | 34 |
| 10 | **+55** | 55 |
| … | **+Fib(n)** | 一般項 |

**定義**: Fib(1)=1, Fib(2)=1, Fib(3)=2, Fib(4)=3, …（§08 §2.3 と同型）。

**カルマ値への副作用**: 各 Δcount 増加は §08 の **Fib ペナルティ**（count n-1→n でカルマ値 **-Fib(n)**）を **逐次適用**。Y02 支払期限切れ（+1）等と **同一月に重なれば両方付与**。

**月の起算**: **取引成立日**（§11.0.1 — 評価確定 or 自動「良い」のタイムスタンプ）を含む月を **第 1 未払い月** とする（境界の時刻・タイムゾーンは詳細設計 **TBD** · §08 U-KRM-01 と整合）。**マッチング日・成約日は起算に使わない**。

**支払い完了**: 当該取引の **8% 未払残 = 0**（全額消込）になった時点で **未払いカウント停止**（以降の月次 Δcount なし）。**部分入金のみでは停止しない**（§11.7.5）。既に付いた Δcount は **戻さない**。

| **生成** | **`deriveTransferCode(userId)`** — `SHA-256(userId)` → **uint24（先頭 3 バイト）** → Base36 大文字 → **4 文字 pad / 6 文字 truncate** → **`U-` + body**（§23 §2.2） |
| **ユーザー確定** | 「**ユーザー ID が一意だから、そこから作る**」（2026-06-07）— legacy `bbs_post_02256` と同型 |
| **スコープ** | **ユーザーごと 1 つ**（`ProjectRules` §6.4）。8% · GMO PT 入金 · P2P 推奨で **同一 `transfer_code`** — **8% と PT で分けない** |
| **振込者名** | 銀行 **振込依頼人名** 欄末尾に追記（例: `ﾔﾏﾀﾞ U-4F9A`）— **コピペ推奨** · GMO **最大 48 半角文字**（[FAQ](https://help.gmo-aozora.com/faqs/6686541cefe214ffd634291a/)） |
| **期待入金** | 既定 `remittance_reference` = 義務者 **`transfer_code`** · 取引は **`trade_ref`** / `note`（`chunk_id`）— §23 §2.4 |
| **衝突時** | 同一ユーザー・同額 pending 複数 → **入金日時 + 義務発生日（`created_at`）で最古 pending に消込**（§23 §2.5.3 · **U-GMO-06 解決**）— 取引サフィックスは **例外のみ** |
| **照合** | **第 1**: `transfer_code` 名義部分一致 + **`amount_yen` 一致** · **第 2**: 複数候補時 **GMO `timestamp` 等の入金日時**（§23 §2.5.2）— legacy は日時なし先頭マッチ（`gmoAozoraReconciliationStore.ts`） |
| **抽出正規表現** | 保存 `^U-[A-Z0-9]{4,6}$` · 名義 `U[\-\－][A-Z0-9]{4,6}`（`rag/bbs_post.csv` **`bbs_post_02252`**) |
| **P2P 推奨** | 当事者間振込でも **同一パターン**（§11.0）— OS 照合は **非対象** |

**TBD**: 合算振込の複数 `trade_ref` 按分（§23 §2.5.5）· 日時照合でも一意化できない場合のサフィックス **例外** regex（§23 §2.5.4）。

#### 11.7.5 部分入金・過入金・貢献費クレジット（ユーザー確定 · 2026-06-07）

> **U-MKT-DSP-07 拡張** · [`23-GMO銀行振込判定.md`](./23-GMO銀行振込判定.md) §3.1 と同期。GMO 照合が **正** — 自己申告のみは **採用しない**（従来方針維持）。

| 区分 | ルール |
|------|--------|
| **部分入金** | 入金額 < 期待 8% でも照合し、**当該取引の積算未払残**を **入金額分だけ減算**。残 > 0 の間は **fee_unpaid 継続** · 月次 Δcount も継続。 |
| **全額消込** | 積算未払残 = **0** で §11.7.2 の **未払いカウント停止**。 |
| **過入金** | 期待額超過分を **貢献費クレジット**（前払い残高）に計上。マイページ等で **残高表示可**（マイナス表記 = 前払いあり）。 |
| **クレジット相殺** | **次回以降**の当該ユーザーに発生する **8% 貢献費** の振込義務額から **クレジットを自動充当**（充当後の不足分のみ銀行振込）。 |
| **返金** | **不可** — 過入金はクレジットのみ（§23 §3.1）。 |
| **製品スタンス** | **推奨しない** — 一括全額振込が正。部分入金は **例外的救済**（しょうがなくそうしている機構 · UI で積極宣伝しない）。 |

**判断・理由**

| 判断 | 理由 |
|------|------|
| **残債のみ減算** | 分割振込の実務に対応しつつ、**義務消滅は全額消込のみ** — カルマ Fib 月次と矛盾しない。 |
| **過入金 = 8% クレジット** | 返金オペなしで **次回貢献費に充当** — 会計が閉じる。 |
| **返金不可** | 返金・争い・監査コスト回避（ユーザー確定）。 |

**TBD**: 合算振込 vs 取引別の按分のみ（§23 §2.5.5）— pending 衝突は **U-GMO-06 解決**（日時 FIFO）。
