## 1. 機能概要

カルマシステムは文明 OS の **統治経済ペナルティ層** である。ユーザー確定モデル（2026-06-07）では **2 つの独立概念** で構成する。

| 概念 | 役割 |
|------|------|
| **カルマ値**（karma value） | 表示・停止判定用スコア。**-100 〜 +100**。登録直後 **0**。 |
| **カルマカウント**（karma count） | 違反・争いイベントの **累積カウンタ**（0 以上整数）。カウント増加のたびに **Fibonacci ペナルティ** でカルマ値が下がる。 |

**時間経過**（月次ルール）でのみカルマ値は **増加** する。争い・取引・裁判イベントは **原則カウントを動かし**、カルマ値は Fib 式または月次ルール **経由でのみ** 変動する（管理者調整・免罪符は例外）。

**永久 BAN**: カルマ値 **-100** 到達 → **ログイン不可**（アカウント削除なし · データ保持 · 他ユーザーはマイページで「永久 BAN」表示）。

**関連機能**: 裁判（§11）· マーケット争い（§06 §11）· プラチナ免罪符（§20）。

## 2. 二層モデル（ユーザー確定）

### 2.1 カルマ値（karma value）

| 項目 | ルール |
|------|--------|
| **初期値** | 新規登録完了後 **0** |
| **レンジ** | **-100 〜 +100**（上限・下限固定） |
| **増加経路** | **時間経過のみ** — §3 月次 +10 ルール（カウント条件を満たす月） |
| **減少経路** | カルマカウント増加時の **Fibonacci ペナルティ**（§2.3） |
| **-100 到達** | **永久 BAN** — ログイン不可。アカウント・R2 データは削除しない。プロフィールは「永久 BAN'd」等として **第三者閲覧可** |
| **直接 ±** | 争い・取引テーブルの数値は **カルマ値を直接いじらない**（§11 · §06 §11 参照） |

### 2.2 カルマカウント（karma count）

| 項目 | ルール |
|------|--------|
| **性質** | カルマ値とは **別カウンタ** |
| **増加** | 裁判・マーケット争い等で定義された **Δcount**（§11 §4 · §6.4） |
| **減少** | ① **毎月 25 日** — 基準日時点で count ≥ 1 なら **count -1**（§3） ② **免罪符** 1 購入 = **count -1**（§4） |
| **副作用** | count が **n-1 → n** に増えた瞬間、カルマ値 **-Fib(n)**（§2.3） |
| **月次 +10 との関係** | 基準月の集計期間中、**count ≥ 1 が 1 日でもある** 場合、その月の **時間経過によるカルマ値 +10 を付与しない**（§3.2） |

### 2.3 Fibonacci ペナルティ（式 · 例）

**定義**: Fib(1)=1, Fib(2)=1, Fib(3)=2, Fib(4)=3, Fib(5)=5, Fib(6)=8, Fib(7)=13, …

**ルール**: カルマカウントが **n-1 から n に増加** したとき、カルマ値は **-Fib(n)** される（同一イベントで複数段増える場合は **各段ごと** に適用）。

**例 A — count 0 から +5 イベント一括**（count → 1,2,3,4,5）:

| 段 | n | -Fib(n) |
|----|---|---------|
| 1→2 | 1 | -1 |
| 2→3 | 2 | -1 |
| 3→4 | 3 | -2 |
| 4→5 | 4 | -3 |
| 5→6 | 5 | -5 |
| **合計** | | **-12** |

**例 B — count 既に 5 の状態から +5**（count → 6,7,8,9,10）:

| 段 | n | -Fib(n) |
|----|---|---------|
| 5→6 | 6 | -8 |
| 6→7 | 7 | -13 |
| 7→8 | 8 | -21 |
| 8→9 | 9 | -34 |
| 9→10 | 10 | -55 |
| **合計** | | **-131** |

**設計意図**: 良いユーザーが大きなミスを 5 回分まとめて食らっても **-12** 程度。悪意ユーザーが同数追加すると **-131** — **累積で悪ユーザーが排除されやすい**（§12.2）。

## 3. 月次サイクル（基準日 **25 日**）

> クレジットカード締め日と同型 — **毎月 25 日** を基準とする。

### 3.1 カウント減衰（25 日）

| 項目 | ルール |
|------|--------|
| **基準日** | 各月 **25 日**（タイムゾーンは詳細設計で確定 · 登録 locale または UTC **TBD**） |
| **条件** | 基準日時点で **カルマカウント ≥ 1** |
| **効果** | **カルマカウント -1**（1 段階のみ · 複数段一括減はしない） |
| **カルマ値** | 減衰そのものでは **カルマ値は増えない**（count 減少は Fib の逆操作ではない） |

### 3.2 時間経過ボーナス（+10）

| 項目 | ルール |
|------|--------|
| **条件** | 集計月において **カルマカウント = 0 の月** を完遂（25 日→翌 25 日前日までの期間で count が常に 0） |
| **効果** | カルマ値 **+10**（上限 +100 でクリップ） |
| **ブロック** | 期間中 **1 日でも count ≥ 1** なら **その月は +10 なし**（「その間カルマは増加しません」= 時間経過増加が停止） |

**legacy との差**: legacy civ-os の `karma_monthly_calm_event_delta`（n-1）+ `karma_monthly_calm_karma_delta`（+10）は **単一軸** 近似。ユーザー確定は **25 日基準** + **count=0 月のみ +10**（§13）。

## 12. 設計判断と理由（ユーザー確定 · 2026-06-07）

> **読み方**: 「判断」= 採用仕様。「理由」= なぜそうするか。「legacy 差分」= civ-os 現行実装・旧 FR との違い。

### 12.1 二層分離（値 vs カウント）

| 判断 | 理由 | legacy 差分 |
|------|------|-------------|
| **カルマ値とカルマカウントを別概念** | 表示・BAN 閾値（値）と違反累積（count）を分離し、**同じ +1 イベントでも累積が進むほど重くなる** Fib を適用できる | legacy は eventCount + fib 累積に近いが **値レンジ・月次・BAN 閾値が未整合** |
| **値の増加は時間経過のみ** | 良い行動は **静かな月** で回復。争いは count 経由でしか値を下げない | legacy 月次緩和は event delta も動かす |
| **-100 で永久 BAN** | 最悪ユーザー排除。データは文明史として残す（R2 INSERT ONLY） | legacy `karma_fib_floor` + 司法停止は **一時停止 + 復帰経路** 寄り |

### 12.2 Fibonacci ペナルティ

| 判断 | 理由 | legacy 差分 |
|------|------|-------------|
| **count 増加 n で -Fib(n)** | 初犯は軽く、累犯は急激に重い。**0→5 で -12、5→10 で -131** の差で悪意排除 | legacy fib 累積式は **同型** だが count/値/停止の **契約が一層に統合されていなかった** |
| **一括 +5 は段ごとに 5 回 Fib** | 単発 -12 vs -131 の例が設計意図の説明可能性になる | `karma_severe_event_bump=5` と **整合可能**（IHL では Δcount +5 として再定義 **TBD**） |

### 12.3 争い・マーケット表記

| 判断 | 理由 | legacy 差分 |
|------|------|-------------|
| **表の数値はすべて Δcount** | カルマ値を争いごとに直接いじると Fib と二重計上になる | 旧 §11「5 回ごと **カルマ -1**」·「カルママイナス 1 カウント」は **値直接減点と混同** |
| **掲示板: 未解決クローズ 5 の倍数 → Δcount +1** | 1 回ごと -1 は過罰。倍数到達で **1 段 count** → Fib で実効ペナルティ | 旧 FR-DSP-09「カルマ -1」→ **ユーザー確定で Δcount +1 に改定**（§11 v2.3） |
| **Y02/Y08 等の「-1」表記** | 旧文書の **-1** は「カルマが下がる方向」の口語。**確定 Δcount は +1**（買い手放置 · 悪意低評価） | §11 §6.4 表を **+1 に統一**（legacy vs 確定 §13 参照） |

### 12.4 月次 25 日

| 判断 | 理由 | legacy 差分 |
|------|------|-------------|
| **基準日 25 日** | クレジット締めと同型でユーザーが予測しやすい | legacy cron 月次は **暦日/30 日** 混在 |
| **25 日に count ≥ 1 → count -1** | 軽い違反は **時間で 1 段ずつ** 薄れる | legacy `karma_monthly_calm_event_delta` と **同型意図** |
| **count ≥ 1 の月は +10 なし** | 「その間カルマは増加しません」— 違反月は **回復なし** | legacy は 30 日以内ペナルティスキップ等 **別条件** |

### 12.5 免罪符

| 判断 | 理由 | legacy 差分 |
|------|------|-------------|
| **プラチナマーケットで count -1** | 経済コストで count を下げる **明示的復帰** | legacy `POST /api/economy/karma/indulgence` と **接続**（IHL 商品名確定） |
| **Fibonacci 価格 · 上限なし** | 月 1 回程度なら **1 PT 前後** · 連続購入は PT 急増で **自己抑制** | 旧「価格 TBD」→ **§22 §5 確定**（U-KRM-05 解決） |
