## 1. 概要・位置づけ

### 1.1 哲学（固定前提）

| 原則 | 内容 |
|------|------|
| **開発者は裁判官ではない** | 運営・開発者が当事者の争いに **判定を下す** モデルは採用しない。 |
| **人間対話が本体** | 国際掲示板（BBS）上の **指摘 → 二人部屋での対話 → 合意または期限切れ** が争い解消の主経路。 |

## 3. 掲示板：指摘フロー

### 3.1 前提（国際 BBS）

- **国際掲示板** を想定する。コンテンツモデレーションは **NG ワードフィルタシステムではない**。
- 問題のある表現は **人間の指摘** と **当事者間対話** で扱う。
- 悪意あるユーザーは **ネスト指摘（マトリョーシカ）** により自ら消耗する設計（自然淘汰）を許容する。

### 3.2 入口 UI

対象の **チャット／投稿** の `...` メニューから、次の 2 操作のみを提供する。

| 操作 | 説明 |
|------|------|
| **引用** | 通常の引用返信 |
| **指摘** | 争いフローを開始（**通報ボタンは置かない**） |

**指摘開始時の必須入力**

1. **タグ選択**（必須）— 指摘カテゴリをタグで選択（タグ一覧は詳細設計で定義）
2. **理由テキスト**（必須）— 自由記述の指摘理由

入力が揃わない限り、二人部屋は開かない。

### 3.3 二人部屋の生成

- 指摘が成立すると、**指摘者** と **指摘されたユーザー** の **二人だけ** が書き込めるボード／チャットが開く。
- **公開観覧**: 他ユーザーは **閲覧のみ**（スペクテーター）。第三者の介入・投稿は不可。
- **重複指摘の禁止**: 同一チャット／投稿について、争いが **指摘中／議論中** の間は **1 人だけ** が指摘できる。他者は同じ対象に重複して指摘を開始できない。
- **非対称な露出**: 悪意あるユーザーは **別々のチャットから複数回** 指摘されうる（チャット横断で +5 と +1 等の **非対称カウント** ロジックはカルマ節 §4.2 参照）。

### 3.4 ネスト指摘（マトリョーシカ）

- 二人部屋の会話中、さらに **指摘** をネストして開始できる。
- **深さ上限なし** — 悪意あるユーザーが自らチェーンを伸ばし消耗する余地を残す（自然淘汰）。
- 各ネストは独立した二人部屋（またはスレッドツリー）として R2 に記録する（詳細設計で構造確定）。

### 3.5 合意フロー

```
指摘 → 二人部屋で対話
  ├─ 合意した場合
  │    1. 指摘された側が「修正後の表現」を提案
  │    2. 指摘者が OK
  │    3. 元の掲示板チャットの **表示** が修正版に切り替わる
  │       （旧版は削除しない · 履歴は R2 に残す · 新しいタイムスタンプが表示勝ち）
  └─ 合意しない場合
       → §3.6 強制クローズへ
```

**合意の自動検知** は **スコープ外**（実装しない）。合意は **明示的な双方操作**（提案 + OK）でのみ成立する。

### 3.6 期限・強制クローズ

| 項目 | 確定値 |
|------|--------|
| **TTL** | 合意に至らない争いは **1 ヶ月** で **強制クローズ** |
| **強制合意** | **行わない** |
| **クローズ後** | 元投稿の表示は変更されない（合意版が無い場合）。カルマ影響は §4 |

## 4. カルマ・プラチナルール

> **カルマ正本**: [`08-カルマシステム.md`](./08-カルマシステム.md) **v2.0** — **カルマ値**（-100〜+100 · 時間経過 +10 のみ増）と **カルマカウント**（Fib ペナルティ · 25 日減衰 · 免罪符）の **二層モデル**。本節は **Δcount イベント源** のみ定義する。争い表の数値は **カルマ値を直接いじらない**。

### 4.1 未解決クローズとカルマカウント（オプション A · ユーザー確定 · v2.3 改定）

- ユーザーごとに **未解決のまま強制クローズされた回数**（**未解決クローズ累計**）を内部カウントする。
- **指摘者・指摘された側は個別に累計** — 同一事件でも **各自の累計** で判定（双方とも未合意クローズ 1 回につき **自分の未解決累計 +1**）。
- **5 回目・10 回目・15 回目…**（5 の倍数到達時のみ）に **カルマカウント +1**（**Δcount +1**）。1〜4 回目・6〜9 回目など **倍数以外のクローズでは Δcount なし**。
- 例: ユーザー X が未合意クローズ 4 回 → カルマ影響なし。5 回目到達で **Δcount +1** → カルマ値は **-Fib(n)**（n は増加後のカウント · §08 §2.3）。10 回目到達で再び **Δcount +1**。
- 合意して終了した争いは、この **未解決クローズ累計** に含めない（詳細設計でイベント種別を確定）。

**legacy vs ユーザー確定**: v2.2 まで「5 の倍数で **カルマ -1**（値直接減点）」と記載していた。**ユーザー確定（2026-06-07）** では **Δcount +1** に改定。実効的なカルマ値減少は **Fib ペナルティ** が担う（§08 §13）。

### 4.2 チャット横断の非対称露出

- **同一チャット／投稿** では open 中 **指摘者は 1 人まで**（§3.3）— 被害者側が同一発言から複数部屋を同時に持つことはない。
- **別チャット** からは **別ユーザーがそれぞれ指摘可能** — 悪行ユーザーは横断的に複数の二人部屋を抱えうる（例: 5 人が 5 チャットから指摘 → 被指摘側に **5 件の open／履歴**）。指摘者側は各チャット **+1** ずつ。
- 未解決クローズ累計は **部屋ごとの強制クローズ** が各自に加算される。**5 の倍数到達時の Δcount +1** は累計に連動するため、横断的悪行は **複数部屋 × 倍数到達** として自然に重くなる（Fib 累積 · §08）。
- **非対称ゴースト救済ルール**（最終発言者で片方だけ免除等）は **採用しない** — §14 参照。

### 4.3 ゾンビ攻撃対策（プラチナ消費）

| 項目 | ルール |
|------|--------|
| **課金単位** | ユーザーあたり **30 回の指摘ごと** に **プラチナコイン 1 枚** を消費 |
| **クールダウン** | **なし** — 支払えば連続指摘も許容（金で買えるがコストは増える） |
| **目的** | 大量自動指摘・ゾンビ攻撃の経済的抑止 |

プラチナ台帳・消費 API は §20（投票・プラチナ）・経済 component と接続する。

### 6.1 取引ステージとプライベートマッチングボード

| 段階 | 内容 |

| 項目 | ルール |
|------|--------|
| **期限** | マッチング後 **2 週間** 以内に振込確認（当事者間で完結） |
| **期限超過** | **出品者（売り手）が取引をクローズ可能** |
| **買い手への影響** | **カルマカウント +1**（§6.4 · §08）、**タグ付き悪評価**（理由必須）、**マイページで他ユーザーが閲覧可能** |
| **連絡不通（Y01）** | 上記 Stage 1 プライベートボード + 2 週間ルールで吸収。別フローは設けない |

### 6.2 争い種別と経路（フロー分岐）

掲示板争い（§3）と **重なるパターン** は §6.6 参照。マーケット固有は下表。

| 種別 | タイミング | 経路 | 記録 |
|------|------------|------|------|
| **A. 双方合意キャンセル** | マッチング後・支払前後問わず双方合意 | プライベートボード上の合意 | **なし** |
| **B. 支払期限切れ** | マッチング後 2 週間・振込未確認 | 売り手クローズ（§6.1） | 買い手: **Δcount +1** · タグ付き悪評価 · 公開プロフィール |
| **C. 配送・支払完了後の争い** | 商品到着・代金・発送が当事者間で OK 後（Y03〜Y07） | **当事者間ディスカッション** → 任意で **公開エクスチェンジ**（コミュニティの力） | **支払完了後のやり取りは常に記録**（取り下げ合意でも **記録は残る**） |
| **D. 出品・出品中の指摘** | 出品ページ・マッチング前後（Y09 他） | **指摘**（掲示板 §3 と同型の入口: タグ + 理由） | 段階別カルマ（§6.4） |
| **E. 評価争い** | 悪意低評価・報復評価（Y08） | **会話ボード**（掲示板二人部屋に準ずる）· **1 ヶ月 TTL** | 公開選択可 · 記録残存 · 評価は **版管理**（新タイムスタンプ勝ち） |

**Y03 / Y04（説明相違・偽物）— 配送完了後**

1. 当事者間で話し合い。
2. **公開エクスチェンジ**（オプション）— コミュニティの力を借りる。
3. **プラチナコイン投票**で正しい側／悪い側を決める（§6.3 · 当事者 opt-in「公開して投票」）。
4. **敗者**: カルマカウント **+5**（悪質扱い）。
5. 出来事は **判例・事例法** として R2 に蓄積 — **参照・引用可能**。
6. 双方が取り下げ合意 → 争いはクローズするが **記録は残る**（掲示板の「合意で表示切替」とは異なる）。

**Y05（配送遅延・未発送）**

- 当事者が話し合って決める。**開発者は介入しない**（「知らん」）。
- カルマカウント **+3**。
- 誠実な謝罪 + 配送業者確認等 → 詐欺でなければ **許される想定**。

**Y06（破損・梱包不良）**

- ディスカッション。**コミュニティ投票あり**（Y03 と同系 · §6.3）。
- カルマカウント **+3**。

**Y07（返品・キャンセル）**

- 「好きにして」— 当事者合意。**カルマカウント +3**。

**Y08（評価の報復・悪意低評価）**

- 悪意低評価を付けた側: **Δcount +1**（マーケット評価軸 · §6.4 · §08）。
- 低評価には **タグ + 理由が必須**。
- 争い → 会話ボード · **1 ヶ月でクローズ**（掲示板 §3.6 と同型 TTL）。
- **いつでも公開可能** — 公開すれば記録・議論は **引用参照可能**。
- 双方取り下げ合意 → **カルマ減点なし** · 評価は **版管理**（新しい時刻が表示勝ち）· **記録は残る**。

**Y09（虚偽出品・画像盗用）— 出品への指摘**

出品に対する **指摘** は次の **3 パターン**（タイミングでカルマが変わる）:

| パターン | タイミング | カルマ | 備考 |
|----------|------------|--------|------|
| 1. 公開 Q&A | 出品中（マッチング前） | （指摘成立時は §6.4 参照） | 質問に答えない → コミュニティは悪いと判断しうる |
| 2. 指摘 | マッチング後・**支払前** | **+1 カウント** | |
| 3. 指摘 | **支払後**（Stage 1 内 · **取引成立前**） | **+5 カウント** | 悪質 |

**Y10（直取引・外部誘導）** — 開発者は介入しない。ユーザーは **システム便益を失う**（血統断絶等 · `06-マーケット` §11.4）。

**Y11（未成年・規制商品）** — 指摘対象外想定（該当商品は出品予定なし）。法令違反時 **Δcount +10**。行政・当局からの指示時は **システム上不使用フラグ**（§6.7 · §14.7.4）。

**Y12〜Y15** — 詳細は [`06-マーケット.md`](./06-マーケット.md) §11.2 解決列。裁判介入は原則 **不要**（指摘・コミュニティ・OSS 文化で足りる）。

### 6.3 プラチナコイン投票（ユーザー確定 · 2026-06-07 · U-MKT-DSP 解決）

> **正本**: §14.7.4 判断・理由。UI 簡素化は legacy `EconomyVote` を参考にする（§20）。

| 項目 | ルール |
|------|--------|
| **適用争い** | 配送完了後の **説明相違・偽物**（Y03/Y04）、**破損・梱包不良**（Y06） |
| **開始** | **当事者の opt-in** — 争いスレッドから **「公開して投票」** を押す（任意 · Y03/Y06 のみ） |
| **投票期間** | **7 日間**（サーバー時刻基準 · 詳細設計で確定） |
| **投票権** | **PT 残高を持つ全ユーザー** — **1 票 = プラチナ 1 枚（1 PT）** 消費 |
| **選択肢** | **二択のみ**: 「売り手が正しい」／「買い手が正しい」 |
| **集計** | **多数決** — 得票が多い側を「正しい側」とする |
| **同点** | **カルマ変動なし**（Δcount 0 · 判例は記録可） |
| **敗者ペナルティ** | 投票終了時に **1 回だけ** **Δcount +5**（Y03/Y04/Y06 共通） |
| **判例化** | 投票結果・議論・集計は **R2 append-only** — **事例法** として参照・引用可能（§6.6） |
| **quorum** | **なし**（最低票数の下限は設けない · コミュニティ参加は任意） |
| **UI** | 争いスレッド **下部** — **2 ボタン + リアルタイム集計表示**（`EconomyVote` 簡素版） |

**フロー（確定メモ）**

```
配送完了後争い（Y03/Y04/Y06）
  → 当事者間ディスカッション（プライベートボード or 公開エクスチェンジ）
  → 任意: 当事者が「公開して投票」
  → 7 日間 · PT 保有者が二択投票（1 票 = 1 PT）
  → 多数決（同点 = Δcount なし）
  → 敗者 Δcount +5（1 回）· 結果を判例 R2 に蓄積
```

## 7. legacy civilization-os から捨てるもの

以下は **IHL rebuild のモデルに含めない**（参照・salvage 時も「採用しない仕様」として読む）。

| legacy 要素 | 捨てる理由 |
|-------------|------------|
| **審理権限ユーザー**（`canResolveJudicialUser`） | 開発者・特権ユーザーが裁判官になる構造 |
| **創世者／L4 管理者による判定・停止バイパス** | 人間審判モデルと矛盾 |
| **reviewStage 1→3・cron 自動段階進行** | 階層審理は採用しない |
| **不服（Appeal）granted/denied の審判官解決** | 当事者対話・合意が本体 |
| **司法信頼（judicialTrust）ランキング** | 本 v2 の指摘モデルでは要求しない |
| **案件タイプ `karma_suspension` / `general` の審理案件** | カルマ停止回復の司法経路は legacy 固有 |
| **`liftSuspension` / 司法停止 API バイパス** | カルマ停止と司法の結合を新モデルに持ち込まない |
| **`/admin/judicial` 運用審理 UI** | 運営審理画面は不要 |
| **通報ボタン・NG ワード自動削除** | 指摘 + 対話モデルと矛盾 |
| **第三者満足度・「争議改訂」ラベル義務** | ユーザー確定で不要 |
| **合意の自動検知・強制合意** | 不可能／スコープ外 |

**salvage してよいもの（参考のみ）**

- R2 append-only イベント記録パターン
- スレッド可視性（公開観覧 / 当事者投稿）の **考え方**
- インボックス・通知の **技術パターン**（仕様は本 v2 に合わせて再定義）

## 14. 設計判断と理由（ユーザー確定 · 2026-06-07）

> **出典**: チャットセッション（`965e82f3-318f-418c-918c-6395adcda701`）におけるユーザー発言。  
> **読み方**: 「判断」= 採用仕様。「理由」= なぜそうするか。「却下した代替案」= AI 提案等をユーザーが明示的に捨てたもの。

### 14.1 全体哲学・スコープ

| 判断 | 理由 | 却下した代替案 |
|------|------|----------------|
| **開発者・創世者は裁判官にならない** | 運営の機嫌・特権判定を避け、**当事者間の合意** を争い解消の本体にする。国際コミュニティで「誰が judge か」論争を生まない | legacy `canResolveJudicialUser`・創世者判定・`/admin/judicial` 運用審理 |
| **争い入口は 2 区分 + バグ** — (1) マーケット **§6 たたき台確定**、(2) 掲示板 **本 v2 主スコープ**、(3) バグ（プラチナ未付与等）は **司法外・修正のみ** | 問題の種類ごとに **最安の解** を選ぶ。バグを司法に載せるとコストと混同が増える。R2 は append-only のため数値修正は余裕 | 万能「general 司法案件」・カルマ停止専用司法経路の継続 |
| **国際掲示板は NG ワード自動モデレーションにしない** | 日本語 OK・英語 NG 等 **文化・言語差** は機械では判定できない。**人の指摘 → 二人部屋対話 → 合意改稿** で扱う | 通報ボタン・NG ワード自動削除・AI 自動制裁 |
| **本設計は「最安・最も効果的・当事者納得」志向**（ユーザー自己評価） | 専任モデレーター・審理階層・自動検知を捨て、**二人部屋 + 期限 + カウント付きカルマ + プラチナ** だけで回す | L4 管理者審理・reviewStage 段階進行・司法信頼ランキング |
| **IHL rebuild · C-Sync 不採用 · R2 ランタイム · GitHub 改善履歴 · UIbuilder は配置・デザイン・紐づけのみ** | 改善議論は GitHub（PR / Discussions / BOARD）、本番争いイベントは R2。争いロジックを Builder 内に書かない | civ-os monolith 継続・C-Sync 4 媒体・UIbuilder で機能開発 |

### 14.2 掲示板 · 指摘フロー

| 判断 | 理由 | 却下した代替案 |
|------|------|----------------|
| **`…` メニュー → 引用 OR 指摘のみ**（**通報ボタンなし**） | 「裏通報 → 運営 BAN」文化を避け、**表に出る指摘** にする | 通報キュー + 運営審査 |
| **指摘はタグ選択（必須）+ 理由テキスト（必須）** | 軽い煽り・匿名通報を抑え、**好き勝手指摘** のハードルを上げる | ワンクリック通報 |
| **指摘成立 → 指摘者と被指摘者だけの二人部屋** | 当事者間で話し合い・合意する **インセンティブ** を作る | 第三者審判官の介入 |
| **ネスト指摘（マトリョーシカ）深さ上限なし** | 悪い表現なら **とことん話し合って疲弊** させ、悪いユーザーは **自然淘汰** させる | 深さ上限・「親 open 中は子 1 件まで」等（ユーザー:「**代替療法**」は悪いと考える） |
| **合意: 被指摘側が修正提案 → 指摘者 OK → 元チャットの表示切替**（削除しない・履歴 R2 保持・**新タイムスタンプが表示勝ち**） | 顔を潰さず直せる。**Time が新しい方が正** と整合 | 投稿削除・運営による書き換え |
| **1 ヶ月合意なし → 強制クローズ** | 国際・async コミュニティ向けに **ゆっくり時間を与える** | 短 TTL・強制合意 |
| **強制クローズ時: 双方それぞれ「未解決クローズ」+1** | 放置も双方にコスト。話し合いで終わらせる方が得 | 片方だけカウント（非対称ゴースト救済） |
| **同一チャット／投稿: open 中は指摘者 1 人まで**（**指摘中／議論中** 表示） | 同一発言への指摘乱立を防ぐ | 同一スレへのクールダウン・複数同時指摘 |
| **別チャットからは別ユーザーが指摘可能**（例: 5 人 × 5 チャット → 被指摘側に 5 件） | 悪行は **横断的に露出** する。指摘者は各チャット +1、被指摘側は複数部屋 | 「5 人から +5 同一チャット」— 同一チャット 1 人制と両立しないため不可 |
| **二人部屋は公開観覧可**（第三者は **閲覧のみ**・介入・判定なし） | 「闇裁判」にしない。第三者の **満足度 UI は不要** | 非公開仲裁・第三者投票 |
| **偽合意・強要合意の自動検知はスコープ外** | **判別不可能**。カルマ回避の OK も本人の自由 | 合意前警告 UI・脅し目的タグ自動検知 |

### 14.3 カルマ · プラチナ

| 判断 | 理由 | 却下した代替案 |
|------|------|----------------|
| **未解決クローズ累計の 5・10・15… 回目到達時のみ Δcount +1**（**オプション A · v2.3**） | v2.2 の「カルマ -1」は **値直接減点**。ユーザー確定二層モデルでは **count +1 → Fib** が正。**1 回ごと Δcount は過罰** | オプション B · 1 回ごと Δcount · **値直接 -1**（legacy 混同） |
| **指摘者・被指摘者は同一事件でも個別累計** | 双方に **話し合いインセンティブ**。片方だけ免除しない | 事件単位の共有カウンタ |
| **マーケット表はすべて Δcount**（+1,+3,+5,+10） | カルマ値直接操作と Fib の **二重計上を防ぐ** · 正本 §08 | 争いごとにカルマ値 ±1 |
| **指摘 30 回ごとにプラチナ 1 枚消費** | **捨てアカウントのゾンビ指摘** を経済的に不可能にする | — |
| **指摘クールダウンなし** | 「**したかったらすればよくない？**」— 抑制は **プラチナ** に任せる | 同一相手・同一スレへのクールダウン |
| **免罪符（黄金ヘラクレス教）で count -1** | 経済コスト付き復帰 · §08 §4 · §20 | カルマ値の直接購入 |

### 14.4 翻訳 · 言語（§21 横断）

| 判断 | 理由 | 却下した代替案 |
|------|------|----------------|
| **新規登録で言語設定（必須）。国籍は不要** | 全画面 UI を登録言語に統一。国コードは争い解決に不要 | 国必須（legacy civ-os オンボーディング） |

|------|------|----------------|

#### 14.7.0 取引成立判定（ユーザー確定 · 2026-06-07）

> **取引成立の定義・Stage 2〜3 · タイムライン正本**: [`06-マーケット.md`](./06-マーケット.md) **§11.0.1**（配送完了 + 評価 · 1 ヶ月自動「良い」· 8% 起算）。本節は **判断・理由のみ**。

| 判断 | 理由 | 却下した代替案 |
|------|------|----------------|
| **取引成立 = 配送完了確認 + 評価確定**（マッチング／`settled` 単独では不可） | 実物が届き問題がなければ取引完了。**成約直後に 8% を課すと未配送・未評価の段階で義務が走る** | オークション決着＝取引成立 · 支払確認＝取引成立 |
| **1 ヶ月無評価 → 自動「評価:良い」** | **1 ヶ月無言 = 問題なかった** とみなす。**無期限 pending**（評価待ちのまま fee・争い境界が曖昧）を防ぐ | 無期限評価待ち · 運営が手動クローズ |
| **8% `fee_unpaid` は取引成立後のみ起算** | 経済義務と **実質完了** を同期（§06 §11.7 · §08 §5.1 · §23） | マッチング直後から 8% クロック |

#### 14.7.1 取引ステージ・マッチング段階

| 判断 | 理由 | 却下した代替案 |
|------|------|----------------|
| **Stage 0〜1** — マッチング後は **プライベートボード** で P2P（振込・発送） | **原始メルカリ** に近い。OS は非エスクロー（`mkg_p2p_core`）。当事者が自分で回すのが最安 | 取引ナビ常時公開 · 運営仲介振込 |
| **プライベートボードは第三者非公開** | 連絡先・振込のやり取りを **全公開しない** | 掲示板二人部屋と同一（公開観覧） |
| **宅配便匿名配送は今は諦める** | 規模がついてから検討。今はコスト対効果が合わない | 初版から匿名配送必須 |
| **双方合意キャンセル → 記録なし** | 気が変わっただけなら **摩擦ゼロ** でよい | すべて trade-event 化 |
| **支払期限 2 週間 · 売り手クローズ可** | 買い手の放置に **明確な期限** を与える（Y01/Y02 一体） | 無期限待ち · 運営が強制クローズ |
| **期限切れ買い手: Δcount +1 + タグ付き悪評価 + マイページ公開** | 他ユーザーが **事前に避けられる** 透明性 · Fib 副作用は §08 | 内部のみペナルティ · カルマ値直接 -1 |

#### 14.7.2 配送完了後・出品指摘・評価（Y03〜Y09 · Y08）

| 判断 | 理由 | 却下した代替案 |
|------|------|----------------|
| **支払完了後の争いは常に記録**（取り下げでも残す） | **判例・事例法**（§6.6）の素材。後続取引の参考になる | 取り下げで DELETE |
| **Y03/Y04: 話し合い → 任意公開 → プラチナ投票 → 敗者 +5** | 開発者判定なしで **コミュニティが正誤の目安** を出す。悪質は重く | 運営鑑定 · 返金保証 |
| **正直取引インセンティブ** — 詐欺 10 万でも **3 回正しい取引の方が得** | 高貴なユーザー層・自然淘汰（§20）で **誠実取引が利益になる** 設計 | 刑罰のみ · カルマのみ |
| **Y05/Y07: 当事者に任せる（「知らん」「好きにして」）+ カルマ +3** | 細部は P2P で足りる。OS は **カウントだけ** 付ける | 発送強制 · 返品義務付け |
| **Y06: +3 + コミュニティ投票可** | 破損は事実争いになりやすい — Y03 と同じ **投票オプション** | 保険・損害賠償代行 |
| **Y09: 出品への指摘 3 段階**（公開 Q&A / 支払前 +1 / 支払後 +5） | タイミングで **悪質度が違う** のをカウントで表現 | 一律ペナルティ |
| **Y08: 低評価はタグ+理由必須 · 悪意評価者 Δcount +1 · 評価争いは 1 ヶ月 TTL** | 報復評価を **理由付きで可視化**。掲示板争いと **同じ時間感覚** · §08 Fib | 評価機能なし · カルマ値直接 -1 |
| **Y08 取り下げ: カルマ減点なし · 評価版管理 · 記録残る** | 掲示板合意と同様 **Time が新しい方が表示** — ただし公開議論は引用可能 | 取り下げで評価削除 |

#### 14.7.3 低優先・スコープ外（Y10〜Y15）

| 判断 | 理由 | 却下した代替案 |
|------|------|----------------|
| **Y10 直取引・外部誘導: 開発者は介入しない** | ユーザーが **システム便益を自ら放棄**（血統断絶）すればよい | DM 監視 · 自動 BAN |
| **Y11: 該当出品予定なし。違反時 Δcount +10。行政指示 → 不使用フラグ** | 個性は排除しない — **法遵守が前提**。開発者は judge ではないが **行政命令には従う**（人間ゲート） | 事前カテゴリ封じ · ハード DELETE |
| **Y12 手数料: 開発者は介入しない。8% は貢献料** | ヤフオクより安い **貢献フィー** として理解すればよい | 手数料仲裁 |
| **Y13 スナイプ: 入札履歴公開 · ユーザーが指摘** | 技術的検知より **透明性 + コミュニティ** | 共謀検知 AI |
| **Y14 追加請求: 指摘される** | 落札後の言い換えは **コミュニティが炙り出す** | 価格ロック強制 |
| **Y15 テンプレ争い: 起きない想定** | **著作権なし · OSS fork 文化** — fork/利用で貢献ポイント | ライセンス司法 |
