## ① 機能概要

貢献度は、いいね・コピー・組み込み等の **Kernel 定義イベント** を換算式で数値化し、レイヤー0〜3 の **ノード単位** に非負で蓄積する経済指標である（PlatinumCoinRules §5.1）。子ノードに評価が付くと **10% が依存グラフ上の祖先へ上流分配** され、貢献度 100 到達ごとにプラチナコイン 1 枚が付与される（§5.2・§5.3）。実装の中核は `contributionEconomy.ts` で、R2 キー `economy/contribution/`・`economy/graph/` に記録する（legacy civ-os は C-Sync 経由 — **IHL では R2 append-only + GitHub 改善履歴**）。§15 の「3% プール内 10%」還元とは **別経路** である。

## ② ユーザーができること

| 操作 | 説明 |
|------|------|
| コンポーネント貢献度確認 | `GET /api/components/:id/contribution-state` — score・更新時刻 |

## ③ スコープ内 / 外

### スコープ内

- 貢献度 Δ 適用・上流 10% 分配（`applyContributionDelta`）
- ノード依存グラフ（`economy/graph/{nodeId}.json`）
- 100 単位 → プラチナ 1 枚（`addScoreAndMaybeMint`）
- 経済イベント R2 記録（legacy: `economy/csync/events/` — **IHL では dispute/karma 同型の append-only イベント列**）
- いいね入力（PlatinumCoinRules §3 — 貢献度入力となりうる）
- ホーム・コンポーネント詳細での **可視化**

### スコープ外

- マーク貢献 3% プールからの還元（PlatinumCoinRules **§15** — 別経路）
- 投票消費・マーケット取引 8%（§14 文明 OS 運営プール）
- レイヤー4（固定資産）へのいいね・貢献（Governance 投票不可）
- 貢献度の **減算**（非負 invariant）
- IHL 内の経済・プラチナ（文明 OS 固有）

## ④ 機能要件（番号付き）

1. **FR-CONTRIB-01** 貢献度 score は **非負** とし、ノード ID ごとに `ContributionState` を R2 に保持する（§5.1）。
2. **FR-CONTRIB-02** 換算イベント（いいね・コピー・組み込み等）は Kernel 実装で列挙し、`eventType` を R2 イベント列に残す（legacy civ-os: C-Sync — **IHL 不採用**）。
3. **FR-CONTRIB-03** `applyContributionDelta` は子に Δ を反映後、祖先がいれば **upstreamPercent（既定 10%）** を重み配分で祖先へ分配する（§5.2）。
4. **FR-CONTRIB-04** 祖先が無い場合、upstream 分は **子に残す**（`contributionEconomy.ts` 単純化 — 意図コメント参照）。
5. **FR-CONTRIB-05** `contributionPerPlatinum`（既定 100）到達ごとに `issueCoin` 1 枚、端数は繰越（§5.3）。
6. **FR-CONTRIB-06** プラチナ上流分配は **投票等で子に付与されたコイン** にも同一グラフ・重みで適用する（§5.2 2項）。
7. **FR-CONTRIB-07** `NodeGraphEntry.ancestors` は `{ nodeId, weight }[]` とし、レイヤー係数・`log(1+n)` 等は spec/`economyMasterStore` で解決可能とする。
8. **FR-CONTRIB-08** 発行理由に「貢献度到達（100 単位）PlatinumCoinRules §5.3」を含める（§8 発行条件）。
9. **FR-CONTRIB-09** ユーザー総貢献は `applyUserContributionDelta` でプロファイル側にも反映する。
10. **FR-CONTRIB-10** API 応答は `nodeId`・`score`・`updated_at` を返し、内部 graph は必要時のみ管理者経路で編集する。

## ⑪ GitHub 開発者貢献度（FR-CONTRIB-GH-*）

> **追記 2026-06-18**: ユーザー要求「GitHub の PR・Issue・コード貢献をプラットフォームの貢献度スコアに反映したい」への対応。IHL rebuild スコープで新設。

### §AI仮定（FR-CONTRIB-GH）

本節の実装数値・閾値はすべて **AI 仮定（草案）**。採用前に人間確認が必要。

| 仮定項目 | AI 仮定値 | 根拠・変更可能性 |
|---------|---------|----------------|
| webhook イベント取得方法 | GitHub App webhook（`pull_request` / `issues` / `push` / `release` イベント） | GitHub API `public` repo は unauthenticated 5000req/h。webhook は即時性があり推奨 |
| ポイント換算サイクル | 受信ごとにリアルタイム Δ 適用 | バッチ（1日1回）でも可 |
| 同一ユーザーの重複防止 | `contribution_event_id = github_delivery_id` でべき等（R2 キーで重複チェック） | GitHub は同一 delivery_id を再送することがある |

### GitHub webhook → 貢献度換算表（§AI仮定）

| GitHub イベント | アクション | 換算 Δ | 備考 |
|----------------|-----------|--------|------|
| `pull_request` | `opened` | +15 | 新 PR 作成 |
| `pull_request` | `closed` + `merged=true` | +30 | PR がマージされた（作者に） |
| `pull_request` | `review_requested` | +5 | レビュー依頼 |
| `pull_request_review` | `submitted` + `state=approved` | +20 | PR レビュー承認 |
| `pull_request_review` | `submitted` + `state=changes_requested` | +10 | レビュー（変更要求もレビューとしてカウント） |
| `issues` | `opened` | +8 | Issue 起票 |
| `issues` | `closed` | +5 | Issue クローズ（担当者に） |
| `issue_comment` | `created` | +3 | Issue / PR コメント |
| `push` | commits 数 × 2 | + (commits × 2) | 直接 push（max +20 per push） |
| `release` | `published` | +50 | リリース発行 |
| `star` | `created` | +1 | スター獲得（リポジトリオーナーに） |

> **上流分配**: 換算後の Δ に `upstreamPercent`（既定 10%）を乗じて依存グラフ祖先へ分配（FR-CONTRIB-03 と同一経路）。
