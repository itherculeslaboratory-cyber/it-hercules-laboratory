## ① 機能概要

データ設計は、文明 OS 全エンティティが **core + rag**（`CoreEntityBase.md`）を満たし、C-USB で接続可能な **文明原子** として扱われることを定義する横断層である。永続真実は R2 の **INSERT ONLY** イベント列（`R2Engine.md`）にあり、状態は replay で再構築する。IHL（個体画像レイク）向けには、同思想の **ファイル契約・schema/manifest/snapshot**（`指示/it-hercules-laboratory/99-アーカイブ/2026.06-06-legacy`）を **shared** として移植する。コンポーネント検索の完全データカタログは fb_00096 等 oral が多く、**統一要件書は partial**。

## ② ユーザーができること（利用者視点）

| 利用者 | できること |
|--------|------------|
| **開発者** | CoreEntityBase 準拠の JSON/API 定義、C-USB 互換 Component 作成、R2 キー規約に沿った INSERT |
| **Builder ユーザー** | Component の core/rag メタを編集し、系譜（parent_uuid / generation）を追跡 |
| **RAG / 検索** | title・description・tags・embedding による横断検索（インデックス設計 §5） |

| **IHL 設計者** | civilization-os の append-only/manifest パターンを **新 repo schema** に写経 |

※ 「データ設計」単体のユーザー画面はなく、各 FeatureNode UI がスキーマを **消費** する。

## ③ スコープ内 / 外

### スコープ内

- `CivilizationEntityBase` = `id` + `kind` + `layer?` + `core` + `rag`
- `CivilizationCoreMeta`（uuid, parent_uuid, ancestor_chain, hashes, generation, timestamps）
- `CivilizationRagMeta`（title, description, tags, embedding 等）
- C-USB 拡張（io, compatibility, security, intent_ref）
- R2 イベント固定構造・SHA-256 ハッシュチェーン
- R2 エンティティ保存パス `/history/entity/{kind}/{uuid}.json`
- IHL: capture / individual / searchable_capture_set / manifest / run_info 等（Phase 1）

### スコープ外

- 常駐 DB を真実源にすること（IHL も R2/Parquet のみ）
- CoreEntityBase **非準拠**の新 API・新 Component
- R2 UPDATE/DELETE（憲法違反）
- 横断 **単一データカタログ MD** の完全整備（現状 scattered）
- 120 画面 legacy カタログへの新規追記（`frontend/spec/legacy/` 参照のみ）

## ④ 機能要件（番号付き）

### 4.1 文明 OS（CoreEntityBase / C-USB / R2）

1. **FR-DATA-01** 新規エンティティは `CivilizationEntityBase` を満たし、`core` と `rag` を必須とする（CoreEntityBase §3）。
2. **FR-DATA-02** Component は `CivilizationUsbPort` として `io`・`compatibility`・`security`・`intent_ref` を持つ（C-USB §2）。
3. **FR-DATA-03** R2 文明史は INSERT ONLY。イベントは `prev_hash`・`event_hash` を必須とする（R2Engine §2–§4）。
4. **FR-DATA-04** 現在状態は保存せず `replay(R2_events)` で再構築可能とする（R2Engine §6）。
5. **FR-DATA-05** RAG インデックスはテキスト系・構造系・ベクトル系の 3 軸で設計する（CoreEntityBase §5）。
6. **FR-DATA-06** 系譜変更は `parent_uuid` / `ancestor_chain` / `generation` で表し、意味的ハッシュで整合を取る。
7. **FR-DATA-07** 変更は 4 媒体同期（spec / post / commit / R2）を経る（ProjectRules — REQ-001）。
8. **FR-DATA-08** ドメイン別拡張（UiComponentEntity, SpeciesEntity, HistoryEntity 等）は `kind` で区別する（CoreEntityBase §4）。

### 4.2 IHL ファイル契約（shared — `指示/it-hercules-laboratory/99-アーカイブ/2026.06-06-legacy`）

9. **FR-DATA-09** 永続保存先は **Cloudflare R2 のみ**。DB 禁止・上書き禁止・削除禁止。
10. **FR-DATA-10** 全 pipeline component は **input manifest** を読み、**output manifest**・**run_info.json**・**errors.jsonl** を必ず出力する。
11. **FR-DATA-11** 全出力に `run_id`・`schema_version`・`created_at`・`pipeline_version`・`input_hash` を付与する。
12. **FR-DATA-12** Phase 1 必須 schema: capture, individual, searchable_capture_set, embedding_manifest, tag_event, usage_event 等（詳細設計書列定義）。
13. **FR-DATA-13** 現在採用版は `manifests/latest/` または **pointer JSON**（方式 B）で表現し、既存 snapshot の上書きで latest を更新しない。
14. **FR-DATA-14** ID は ULID 系 prefix（`ind_`, `cap_`, `run_`, `snap_` 等）で生成する。
15. **FR-DATA-15** タグは固定列ではなく **append-only tag event JSONL** と aggregate ビューで表現する。

## ⑤ 非機能要件

1. **NFR-DATA-01** `content_hash` / `semantic_hash` / `lineage_hash` の算出規則をドメインごとに文書化し、再計算可能とする。
2. **NFR-DATA-02** schema 変更は **version 付き新 schema** のみ。既存 Parquet の in-place ALTER 禁止。
3. **NFR-DATA-03** IHL component は `exists` チェック後 write — 衝突時例外（AI実装指示書 完了判定 #3）。
4. **NFR-DATA-04** 大規模 embedding は `.npy` + manifest offset 参照（メモリ効率・再解析可能）。
5. **NFR-DATA-05** civilization-os と IHL の **キー名前空間** を混在させない（`world/` vs `raw/`/`manifests/`）。

## ⑥ MiniKernel / C-USB 上の位置づけ

| 層 | 役割 |
|----|------|
| **WorldSystem** | World → FeatureNode → Kernel → Component → SubComponent — **データ設計は全層に横断** |
| **Component** | 文明原子 = C-USB 1 件 = CoreEntityBase + io |
| **Kernel** | R2 replay・検索 Kernel・manifest builder Kernel |
| **R2 (K)** | 物理法則 — INSERT ONLY エンジン |
| **Builder (B)** | Component JSON 編集が core/rag を直接触る UI |
| **IHL components** | ingest / thumbnail / embedding / manifest builder = **C-USB Driver 候補**（外部技術を file 契約に変換） |

データ設計は **機能番号というより憲法層** — MiniKernel 上のすべての IN/OUT がここに従う。

## ⑦ IHL repo との関係

**shared（schema 思想）**

| 移植する | 移植しない |
|----------|------------|
| append-only / immutable / snapshot | Twin / Builder / 120 画面 API |
| schema YAML + dictionary enum | civilization-os `users/*.json` 形状 |
| run_info / output manifest 契約 | プラチナ・カルマ R2 キー |
| Parquet 検索 + embedding locator | CoreEntityBase の **全 kind** コピー（必要 kind のみ） |

IHL Phase 1 必須: R2 ディレクトリ bootstrap、ingest→manifest→Streamlit UI（`00-AI-HANDOFF-BRIEF.md` §7）。

civilization-os は将来 **consumer**（FeatureNode 経由で IHL manifest を読む）になりうる。
