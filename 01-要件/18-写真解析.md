## ①機能概要（平易に3-5文）

写真解析は、個体観測画像から種候補・形態特徴・タグ・taxonomy を導き、ユーザー確定後に R2 へ append-only 保存する機能群である。civilization-os では固体観測 API（`solidObservation.ts`）がヒューリスティック＋任意 CLIP＋Vision HTTP/OpenAI で種候補を提示するが、**種の自動確定は行わない**。IHL（`指示/it-hercules-laboratory/99-アーカイブ/2026.06-06-legacy/`）では thumbnail・embedding（DINOv2）・QC・searchable capture set をパイプラインコンポーネントとして R2 上に構築する思想が詳細化されており、本機能は **civ-os 実装**と **IHL 将来パイプライン**の接点である。

## ④機能要件（番号付き）

### FR-18-01 画像取り込み・解析（civ-os）

- `POST` 画像アップロード（jpeg/png/webp、25MB 上限）または R2 キーから解析。
- `analyzeSolidObservationImage` が色・bbox・換算長さ等の `SolidImageAnalysis` を返す。
- mm 系は scale template / scale sheet と連動。

### FR-18-02 種候補提案（propose-species）

- 処理順: ヒューリスティック → HF CLIP（任意）→ HTTP Vision / OpenAI Vision（`SOLID_VISION_BACKEND`）。
- `SOLID_VISION_SPECIES_ASSIST=off` で HTTP/OpenAI のみ停止。CLIP はトークンがあれば継続。
- レスポンスに `visionAssist` / `clipAssist` 状態を含め UI に表示。

### FR-18-03 ユーザー確定・taxonomy

- 候補は参考順のみ。亜種以降はユーザーが確定。
- R2 種カタログ: ページング・一括 import（REQ-015）。
- GBIF / Wikidata ドラフト連携（API 経由）。

### FR-18-04 LabelMe → ScreenDef

- LabelMe テンプレ JSON から ScreenDef 自動 UI（REQ-011 implemented）。
- fork v2 `labelme_template` で系譜管理。

### FR-18-05 再解析 manifest

- `buildSolidObservationReanalysisManifest`: imageR2Key, scaleTemplateId, bpcmsEngineProfile 等。
- 画像バイナリは manifest に含めない。検証者は R2 から再取得。

### FR-18-06 thumbnail 生成（IHL 契約 — `2026.06,06` §8.2）

- 入力: ingest 後の capture manifest。
- Pillow: EXIF transpose、長辺 512px、JPEG 出力。
- 出力: `derived/thumbnails/.../{capture_id}.jpg`、thumbnail manifest parquet、run_info、errors。
- 失敗行は errors へ追記し処理継続。

### FR-18-07 embedding 生成（IHL 契約 — §8.3・AI実装指示書）

- モデル backend: `dummy`（Phase 1 最小）/ `dinov2`（本番想定）。
- 出力: `embeddings.npy`（float32、NaN 禁止）、embedding manifest（vector_offset, embedding_dim, normalized_flag=true）。
- L2 正規化必須。既存 key への put 禁止（append-only run）。

### FR-18-08 QC（IHL 契約 — 詳細設計書・要件定義1）

- `qc builder`: blur / exposure / scale / background / occlusion の簡易判定。
- 出力: `derived/qc/qc_v1/run_id=.../qc_results.parquet`。
- searchable capture set に `qc_flag`（usable / warning / reject / unchecked）、`qc_score` を載せる。
- 検索 UI で QC フィルタ可能（Phase 1 Streamlit sidebar）。

### FR-18-09 manifest builder・searchable capture set

- captures + thumbnail manifest + embedding manifest を Polars join。
- 出力: `searchable_capture_set` parquet、embedding locator、individual master。
- latest は pointer 方式（上書き禁止）。snapshot_id で版管理。

### FR-18-10 類似検索（IHL Phase 1）

- query capture の embedding file + offset から cosine。
- final score 例: `0.7*cosine + 0.3*size_similarity`（horn/body/thorax、欠損時 0.5）。
- FAISS は Phase 1 不要。subset numpy cosine で可。
- query 自身を最上位から除外する設定をサポート。

### FR-18-11 タグ・usage（IHL）

- tag event: JSONL append-only（invert / review_needed / deprecate 可）。
- tag aggregator → strong / weak / disputed 状態。
- usage event: search / view / similar_search 等を JSONL 記録。

## ⑥MiniKernel/C-USB上の位置づけ

```
FeatureNode(observation/solid) → Kernel(process/view) → Component(観測セッション) → 画像解析 SubComponent
```

| 要素 | 位置づけ |
|------|----------|
| **API** | `backend/src/api/routes/solidObservation.ts` — X（Connector）＋ G（Gnosis: Vision/CLIP/LLM）。 |
| **R2** | セッション JSON、画像バイナリ、種カタログ、スケールテンプレ — K 層の永続法則（INSERT ONLY）。 |
| **IHL コンポーネント** | ingest_normalize / thumbnail_builder / embedding_builder / manifest_builder は C-USB 的な **入出力契約**を持つ CLI（`run.py`）として設計。 |
| **CoreEntityBase** | 画像メタ・taxonomy・観測タグは core + rag を満たす。 |
