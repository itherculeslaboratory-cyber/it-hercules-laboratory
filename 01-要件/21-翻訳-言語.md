## 1. 概要

IT Hercules Laboratory（IHL）および CivilizationOS 移行後の製品全体において、利用者が選んだ **表示言語（locale）** に基づき **UI 文言を統一** し、国際掲示板（BBS）・二人部屋（§11 裁判）では **ユーザー生成コンテンツ（UGC）を閲覧者の言語へ翻訳表示** する横断要件である。

**国籍・居住地・タイムゾーンとは独立** する。言語設定は **UI と翻訳表示の基準** のみを決め、法的管轄や KYC とは結びつけない。

---

## 2. 目的

| # | 目的 |
|---|------|
| 1 | **国際 BBS** を成立させる — 日本語で「問題ない」表現が英語圏で不適切になりうる文脈を、当事者対話（§11）と併せて **言語壁なく** 扱えるようにする |
| 2 | **初回登録から一貫** — オンボーディングで言語を必ず選び、以降の全 UI がその言語で表示される |
| 3 | **設定で変更可能** — 利用者が後から表示言語を切り替えられる（再ログイン不要が理想 · 詳細設計で確定） |
| 4 | **サーバー負荷とプライバシーの分離** — 二人部屋・掲示板本文の **リアルタイム翻訳はクライアント側計算** を第一選択とし、原文は R2 に **翻訳せず** 保存する |
| 5 | **IHL 思想との整合** — ランタイムデータは R2 append-only、改善履歴は GitHub、UI 配置は UIbuilder のみ。C-Sync 4 媒体は **不採用** |

## 3. スコープ

### 3.1 スコープ内

| 領域 | 内容 |
|------|------|
| **アカウント言語** | ユーザー `locale`（BCP 47 想定 · 初期セットは詳細設計）の **登録・保存・変更** |
| **UI 全文言切替** | アプリ内の **製品 UI 文言**（メニュー・ボタン・空状態・エラー・ヘルプ等）を選択言語で表示 |
| **UGC 翻訳表示** | 掲示板投稿・二人部屋メッセージ・引用ブロック等を **閲覧者の登録言語** で翻訳表示（原文併記は UI 設計で決定） |
| **横断接続** | §03 新規登録 · §12 設定 · §07 掲示板 · §11 裁判 · §02 利用規約（多言語条項）との契約 |

### 3.2 スコープ外（本 v1）

| 項目 | 扱い |
|------|------|
| **国籍・居住地による言語の自動推定** | **採用しない** — 国フィールド（`country`）と `locale` は **独立** |
| **サーバー側の公式翻訳キャッシュ（全量）** | 第一選択ではない。必要なら ADR で **オプション** として検討 |
| **翻訳品質の第三者保証・公証** | **要求しない**（§11 と同旨） |
| **利用規約条文の法務翻訳確定** | 機能 #2 の **人間ゲート**（本機能は技術要件のみ） |
| **マーケット争いフロー内翻訳** | §11 §6 プレースホルダ — マーケット確定後に追記 |
| **legacy civ-os の即時実装** | 設計ゲート通過まで **コード変更禁止** |

## 4. 登録時言語設定

### 4.1 原則

- **新規ユーザー** はオンボーディング（§03）で **表示言語を必須選択** する。
- 選択値はユーザープロフィールの **`locale`** フィールドに保存する（R2 `users/{userId}.json` · append-only 更新は新 snapshot または PATCH 許可フィールド · 詳細設計で確定）。
- **国籍（`country`）は言語の代理変数にしない**。国と言語は別入力・別保存。

### 4.2 機能要件（登録）

| ID | 要件 |
|----|------|
| **FR-I18N-REG-01** | オンボーディング完了前に **`locale` が未設定のまま `onboardingComplete: true` にしてはならない**（§03 `FR-REG-06` と整合）。 |
| **FR-I18N-REG-02** | 言語選択 UI は **明示的な一覧**（select 等）とし、ブラウザ `Accept-Language` は **初期値の提案** に留め、無断確定しない。 |
| **FR-I18N-REG-03** | 初回保存経路は `POST /api/auth/setup-profile` の `locale`（§03 正経路）。レガシー `POST /api/auth/locale` は **設定変更** 用として位置づけ直す（詳細設計）。 |
| **FR-I18N-REG-04** | オンボーディング画面自体のラベルは、選択前は **フォールバック言語 1 種**（既定 `ja` または `en` · ADR）で表示し、保存後は **選択言語** で全文切替する。 |

## 5. 設定画面での変更

### 5.1 原則

- 認証済みユーザーは **設定（§12）またはプロフィール相当画面** から `locale` を **いつでも変更** できる。
- 変更は **即時または次ナビゲーション** で UI 全体に反映する（詳細設計で確定）。
- 言語変更は **監査イベント** として R2 に記録してよい（INSERT ONLY · 任意 · プライバシー §10 参照）。

### 5.2 機能要件（設定）

| ID | 要件 |
|----|------|
| **FR-I18N-SET-01** | `/me/settings` またはマイページに **「表示言語」** 項目を設け、オンボーディングと **同一の選択肢集合** を提供する。 |
| **FR-I18N-SET-02** | 言語のみの更新 API（例: `POST /api/auth/locale` または preferences PATCH）を **設定画面の正経路** とする。 |
| **FR-I18N-SET-03** | 保存成功・失敗を **1 行理由** で表示する（`ui-reference/preferences.md` · 空状態ルール準拠）。 |
| **FR-I18N-SET-04** | 言語変更後、**再ログインを要求しない**（理想要件 · セキュリティ上必要なら例外を ADR に記載）。 |

## 6. UI 全文言切替

### 6.1 対象

| 区分 | 扱い |
|------|------|
| **製品 UI** | メニュー・ボタン・フォームラベル・バリデーション・toast・空状態・ローディング・ヘルプ導線 — **すべて** 登録 `locale` に従う |
| **UGC 本文** | §7 — 翻訳 **表示**（保存は原文） |
| **法務条文（ToS）** | 採用言語版の **正本リンク** を表示。機械翻訳の ToS 本文は **正本にしない**（§02 人間ゲート） |
| **システム生成メール** | マジックリンク等 — **未確定**（§11）。理想は `locale` に合わせたテンプレ |

### 6.2 機能要件（UI）

| ID | 要件 |
|----|------|
| **FR-I18N-UI-01** | 認証済みセッション中、**ルート遷移を問わず** UI 文言の基準は **サーバ保存の `locale`** とする（クライアントだけの一時切替は **開発用以外禁止**）。 |
| **FR-I18N-UI-02** | 未対応 `locale` 要求時は **フォールバックチェーン**（例: `en-US` → `en` → `ja`）を適用し、空白文言を出さない。 |
| **FR-I18N-UI-03** | UI 文言リソースは **キー化** し、ハードコード日本語の新規追加を **設計ゲート後の実装規約** で抑制する。 |
| **FR-I18N-UI-04** | UIbuilder（§16）が配置する Component も **同一 i18n 契約** を継承する（Builder は文言 invent しない · 登録済みキー参照）。 |

## 7. 掲示板・二人部屋での翻訳（クライアント側）

### 7.1 背景（国際 BBS × §11 裁判）

- 掲示板（§07）は **国際 BBS** を想定する。
- 指摘・二人部屋（§11）は **当事者の登録言語が異なる** 場合が通常あり、**対話可能性** が争い解消の前提となる。
- 日本語で許容されうる表現が、英語圏の利用者にとって **指摘対象** になりうる — 翻訳は **理解のため** であり、**法的判定の自動化ではない**。

### 7.2 アーキテクチャ原則

```
R2（正本）          クライアント（閲覧者端末）
──────────          ────────────────────────
原文 UTF-8 保存  →  locale_A 利用者が閲覧
（翻訳なし）          → Transform: クライアント翻訳
                      → 表示: locale_A 向けテキスト
```

| 項目 | ルール |
|------|--------|
| **保存** | 投稿・メッセージは **作者が書いた言語の原文** のみ R2 に append |
| **翻訳実行場所** | **閲覧者端末**（Web Worker / WASM / ブラウザ API / オンデバイスモデル等 — **ADR で選定**） |
| **サーバー翻訳** | 第一選択 **ではない**。帯域・ログ・プライバシー理由でクライアント優先 |
| **二人部屋** | 当事者それぞれの `locale` に合わせ **各自の画面** で翻訳表示（§11 `FR-DSP-11`） |
| **公開観覧** | スペクテーターも **自分の `locale`** で翻訳表示 |

### 7.3 機能要件（UGC 翻訳）

| ID | 要件 |
|----|------|
| **FR-I18N-UGC-01** | 掲示板スレ・二人部屋メッセージは **原文言語タグ** をメタデータに持てる（作者 `locale` のスナップショットで可 · 詳細設計）。 |
| **FR-I18N-UGC-02** | 閲覧者 `locale` ≠ 原文言語のとき、**クライアント側** で翻訳を生成し表示する。 |
| **FR-I18N-UGC-03** | 翻訳失敗時は **原文表示 + 短い理由**（オフライン・モデル未読込等）とし、空白にしない。 |
| **FR-I18N-UGC-04** | **合意版**（§11）の表示切替時も、表示層は閲覧者 `locale` で翻訳してよい。R2 上の合意版正本は **原文のまま**。 |
| **FR-I18N-UGC-05** | 指摘理由・タグ説明等、争いフロー UI も **同一翻訳契約** に含める。 |

## 8. マーケット等への横展開

| 機能 | 現時点 | 将来 |
|------|--------|------|
| **§06 マーケット** | 出品タイトル・説明は **UGC 同様** にクライアント翻訳表示を **適用可能**（詳細設計） | 争いフロー（§11 §6）確定後に **商品説明・チャット** を明示 |

| **§09 論文** | 論文タイトル・要約 — **言語混在** 前提。翻訳は閲覧補助 | 学術正本は原文優先表示を ADR |
| **通知** | プッシュ・メール本文 — **locale テンプレ** 未確定 | §12 · 通知 Feature と接続 |

**横断ルール**: 新 FeatureNode で **人間が読む自由テキスト** が出る箇所は、本機能の **UGC 翻訳契約** をデフォルト継承とする。

## 9. component / R2 / GitHub / IHL 思想

| 層 | 翻訳・言語の位置づけ |
|----|----------------------|
| **R2（ランタイム）** | `users/{id}.json` の `locale` · 投稿原文 · 争いイベント — **append-only**。翻訳結果の **一括キャッシュを R2 正本にしない**（クライアント生成が原則） |
| **GitHub（履歴）** | i18n キー追加・翻訳 SDK 選定・ADR — PR / BOARD.md。C-Sync **不採用** |
| **UIbuilder（§16）** | 配置のみ。文言キーは **登録済み catalog** から選択 |
| **Component（C-USB）** | `i18n-provider`（UI）· `ugc-translate-view`（Transform）· `locale-picker`（IN）等として分解候補 — [`../component分解/00-マスターcomponent分解表.md`](../02-設計/_横断/component/00-マスターcomponent分解表.md) 追記は設計確定後 |
| **Kernel** | `locale` 解決・メタデータ付与は **auth / user-preferences** Kernel と接続 |
| **IHL rebuild** | 単一 repo · R2 本番 · legacy civ-os の `OnboardingFlow` locale フィールドは **salvage 参考** |

## 12. 非機能要件

| ID | 要件 |
|----|------|
| **NFR-I18N-01 パフォーマンス** | 初回翻訳モデル読込は **進行表示** を出す。長スレは **段階的翻訳**（viewport 優先）を許容。 |
| **NFR-I18N-02 オフライン** | オフライン時は **原文表示** とし、翻訳不能理由を 1 行表示。編集・投稿は別 Feature のオフライン方針に従う。 |
| **NFR-I18N-03 プライバシー** | クライアント翻訳は **第三者 SaaS 送信を最小化** する。外部 API 利用時は **明示同意・データ経路** を ToS / プライバシー（§02）と整合。 |
| **NFR-I18N-04 アクセシビリティ** | 言語切替はキーボード操作可能。翻訳テキストも **スクリーンリーダー** が読める DOM を維持。 |
| **NFR-I18N-05 セキュリティ** | 翻訳結果を **XSS 入口** にしない（サニタイズは表示層の既存ルールに従う）。 |
| **NFR-I18N-06 保守** | 文言キー欠落は CI で検知（設計確定後）。ユーザー向けに「未実装」文言は出さない。 |
