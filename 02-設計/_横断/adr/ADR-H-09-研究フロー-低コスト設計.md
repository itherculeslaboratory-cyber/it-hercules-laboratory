# ADR-H-09: 研究フロー — 低コスト設計（観測→仮説→試す→記録→引用 を最短に）

## 文脈（ユーザーの問い）

> 「論文タブ＋ケース分けって、**本当に研究を進める一番安いルート**？ 進行中論文・温度ケース・並行検証は OK だけど、**変な粘り（無駄な引っかかり）は削って**。**観測→仮説→試す→記録→引用 が最短クリック**になるように批判的に見直して。」

研究の価値は **血統の証明 + 再解析可能な研究履歴の蓄積**（アイディア §1）。中心は画像ではなく **Cross・Measurement・Observation・Content**。論文は **終点ではなく起点**（アイディア §3）。

→ 本 ADR は **「論文タブ＋ケース」案を批判的にレビュー**し、研究速度（velocity）を最大化する **主導線（primary path）**と、**作り込みすぎない（over-engineer しない）境界**を確定する。

---

## 1. 批判的レビュー — 論文タブ＋ケースは最安か？

### 結論（正直な評価）

**「論文タブ＋ case チップ」は IA（情報設計）としては正しいが、それ単体では研究 velocity の最安ルートではない。** case 分けは **後から探す（検索・分類）コストを下げる**が、**研究を前に進める（観測→記録→引用）コストは下げない**。velocity を律速するのは **タブ構造ではなく「観測データを論文に運ぶ手間」と「途中状態を保存できるか」**。

| 論点 | 論文タブ＋ケースで足りるか | 不足（＝変な粘り） |
|------|:--------------------------:|--------------------|
| 後から研究を**探す/分類** | ◎ 足りる（[ADR-H-07](./ADR-H-07-掲示板-入口4つ-論文内研究.md)） | — |
| 観測を論文へ**運ぶ** | △ 不足 | 観測 JSON を手コピー／別画面往復（[`09`](./09-論文.md) §11 現状: localStorage PoC のみ） |
| **途中状態**（進行中・並行ケース）を残す | △ 不足 | Content に `status=in_progress` と複数ケースを **1 つの起点**で持てない |
| 仮説→試す→**結果記録**のループ | ✗ 不足 | 現状 Paper Match は「不足キーを見る」止まり（[`09`](./09-論文.md) §11.3） |
| **引用**（Cross/観測/他論文へ） | △ 不足 | Citation がワンクリックでない |

**削るべき「変な粘り」（velocity を殺す UI 摩擦）**:

1. **画面往復**: 観測 → 別タブで論文 → また観測、の往復。→ **観測詳細から「この観測で記録を始める」1 ボタン**で解消。
2. **保存できない途中状態**: 進行中・温度ケース・並行検証を「下書きが消える」恐怖で書けない。→ **Content は append-only。`in_progress` が一級市民**（消えない）。
3. **論文＝完成品の重圧**: 「論文」という語が完成を連想させ、起点として書き始めにくい。→ **「研究ノート（research_note）→ 仮説（hypothesis）→ 論文（paper）」を同じ Content の status 遷移**にし、**最初の 1 クリックを軽く**する。
4. **引用の手作業**: 引用先 ID を手で貼る。→ **観測/Cross/論文の各画面に「引用に追加」**を常設し Citation を自動生成。

→ **論文タブ＋ケースは残す（探索の正解）。その上に「運ぶ・残す・回す・引く」を主導線として足す**のが最安。

## 2. 主導線（primary path）— 最短クリック

> **観測 → 仮説 → 試す → 記録 → 引用** を、画面往復ゼロに近づける一本道。

```text
Individual（個体）
   └─[個体を選ぶ]→ Capture（撮影・取り込み）
        └─[計測を足す]→ Measurement / Observation（数値・状態）
             └─[交配を記録]→ Cross / OffspringAssignment（血統 Truth）
                  └─[研究を始める]→ Content（research_note → hypothesis → paper）
                       ├─ status=in_progress（進行中・温度ケース・並行検証はここに同居）
                       ├─[試す]→ 追加観測 → 再計測（ループは Capture/Measurement に戻るだけ）
                       ├─[記録]→ 結果を Content に append（消えない）
                       └─[引用]→ Citation（Cross / 観測 / 他 Content / Template へ）
```

**設計原則**:

- **同じ実体の status 遷移で完結**: `Content.content_type`（research_note / hypothesis / replication_report / paper / review / article / research_gap）と `status`（proposed → under_review → testing → supported/rejected ＝アイディア §3 の仮説状態）。**新しい画面を増やさず status を進める**。
- **進行中が一級**: `in_progress` の Content と **並行ケース**（温度実験 A/B/C など）は **1 つの研究ノートにぶら下がる子記録**として持つ（append-only。消えない・あとで paper に昇格）。
- **どの画面からでも研究を始められる**: 観測詳細・Cross 詳細・検索結果に **「研究を始める / 記録を追加」** を常設。

## 3. クリックバジェット（研究タスク別・3 クリック原則）

> ホーム or 該当画面起点で、研究を **前に進める**操作の最短クリック数目安（[`02-設計/_ui-global/00-画面一覧-全体像.md`](../02-設計/_ui-global/00-画面一覧-全体像.md) の 3 クリック原則に整合）。

| 研究タスク | 最短クリック | 経路 |
|------------|:-----------:|------|
| 観測を**記録**する（計測 1 件追加） | **2** | 個体詳細 →〔計測を追加〕→ 保存 |
| 観測から**研究を始める**（research_note 起票） | **1** | 観測詳細 →〔研究を始める〕（status=in_progress で即起票） |
| **仮説**に進める | **1** | 研究ノート →〔仮説にする〕（content_type=hypothesis） |
| **試す**（並行ケース追加） | **2** | 研究 →〔ケースを追加〕→ 条件入力（温度等） |
| 試した**結果を記録** | **2** | 研究 →〔記録を追加〕→ 数値/メモ append |
| **引用**を足す（観測 / Cross / 論文） | **1** | 各画面の〔引用に追加〕 |
| 論文として**公開** | **2** | 研究 →〔論文にする(paper)〕→ 確認 |
| 後から**探す**（case で絞る） | **2** | 掲示板 › 論文 → case チップ（ADR-H-07） |

**合計（観測 1 件 → 引用付き進行中論文まで）**: おおむね **7〜9 クリック**で **下書き消失なし**に到達。往復・コピペ・保存不安をゼロにするのが目標。

## 4. 作り込みすぎない境界（NOT over-engineer）

> velocity を上げるのは「運ぶ・残す・回す・引く」。下記は **Phase 1 では深追いしない**（やると粘りになる）。

| 機能 | 判断 | 理由 |
|------|------|------|
| **Research Dashboard**（paper/review/replication/citation/gap/再現性スコアの集計画面） | **Phase 1 は最小 or 後回し** | Dashboard は **派生 Snapshot の表示**（[`ADR-H-04`](./ADR-H-04-設計規約-v1.2.md) §8）。研究を進めはしない。プロフィールの貢献度内訳（[`ADR-H-08`](./ADR-H-08-指標とドメイン仕分け.md)）で代替可。**フル集計 UI は Phase 2** |
| **Knowledge Graph**（ノード/エッジ可視化） | **Phase 2/3**（アイディア §1 Phase3 と整合） | 派生成果物。Truth（Content/Citation）が貯まる前に作っても空。**先に引用を貯める** |
| **論文 embedding gap / missing perspectives**（[`09`](./09-論文.md) FR-PAPER-09/10） | **Phase 2 以降** | 価値は高いが velocity の律速ではない。論文母数が要る |
| **テンプレートエンジン（美学テンプレ 5 種）**（FR-PAPER-04） | **構造のみ Phase 1・装飾は後** | sections/conditions/state の **構造**は要るが、文体スキンは velocity に無関係 |
| **Reproducibility score 自動算出** | **Snapshot・後追い** | 追試イベントが貯まってから再計算すればよい（Truth は `replication_report`） |
| **査読ワークフロー本実装** | **Phase 1 は BBS 論文板で代替**（ADR-H-07） | 独立 WF は重い。case=review ＋ Citation で足りる想定 |

**逆に Phase 1 で必ず要る（velocity の核）**:

- `Content`（research_note/hypothesis/paper…）＋ `status` 遷移（append-only）。
- **進行中 + 並行ケース**の保持（消えない子記録）。
- **観測/Cross → 研究 への 1 ボタン導線**（運ぶ）。
- **Citation ワンクリック**（引く）。

## 5. 研究 UX 評定（1 段落・ユーザー回答）

**論文タブ＋ case 分けは「探す/分類」の正解だが、それだけでは研究速度の最安ルートにならない。** velocity を律速するのは **タブ構造ではなく「観測を論文へ運ぶ手間・途中状態を恐れず保存できるか・仮説→試す→記録→引用を往復なく回せるか」**。したがって本 ADR は **case タブを維持しつつ**、(1) 観測/Cross 詳細から **1 クリックで研究起票**、(2) `Content` を **append-only・`in_progress` を一級**にして進行中/温度ケース/並行検証を消えない子記録で持つ、(3) **Citation を各画面ワンクリック**、の 3 点を主導線に据える。**Dashboard・Knowledge Graph・embedding gap・査読 WF は Phase 1 で深追いしない**（派生 Snapshot・母数依存で velocity に効かない＝変な粘り）。これで「観測→仮説→試す→記録→引用」は **往復ゼロ・下書き消失なし・概ね 7〜9 クリック**で一周でき、研究履歴が append-only に貯まる。
