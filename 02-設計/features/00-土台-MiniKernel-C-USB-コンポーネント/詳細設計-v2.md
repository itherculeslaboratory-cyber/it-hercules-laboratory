## 2. データ契約（schema YAML · R2 キー · append-only 規約）

### 2.1 schema-pack v1（契約の唯一の正本）

`02-設計/_横断/schema/` が IHL の**入出力契約の canonical**。component の `manifest.yaml` は `in_schema_ref` / `out_schema_ref` で**参照するだけ**（列定義を複製しない · ADR-Phase2 §6）。

| schema 群 | 役割 | 代表ファイル |
|-----------|------|-------------|
| `manifest/` | component 入出力 manifest | `input_manifest` · `output_manifest` · `run_info` · `errors` |
| `capture/` | 観測 Truth | `capture` · `individual` · `measurement` |
| `events/` | append-only イベント（Truth） | `tag_event` · `annotation_event` · `label_event` · `preference_event` |
| `economy/` | IHL 経済（独立 schema） | `pt_event` · `supporter_event` · `market_economy_event` |
| `governance/` | 統治 | `vote_event` · `dispute_event` |
| `snapshots/` | 再計算可能な投影（Truth ではない） | `individual_master` · `tag_aggregate` · `*_summary` |
| `common/` | 共通断片 | `provenance` · `core_subset` |
| `dictionaries/` | enum 正本 | `value_origin` · `measurement_name` · `measurement_method` ほか |

**契約必須メタ（各 `*.schema.yaml`）**: `$schema_name` · `schema_version`（整数）· `required` · `properties`（列→型→`enum_ref`）。enum はベタ書きせず `dictionaries/*.yaml` を `enum_ref` で参照。`SchemaRegistry._resolve_node` が `enum_ref` を実 enum 配列へ展開する（`schema_validator.py` L146–165）。

### 2.2 R2 キー設計（append-only）

| 系統 | プレフィックス例 | 書込規約 |
|------|------------------|----------|
| raw | `raw/<source>/<run_id>/...` | no-overwrite |
| normalized | `normalized/captures/<run_id>/part-<uuid>.parquet` | 新規キーのみ |
| derived | `derived/embeddings/<run_id>/emb_*.npy` | 新規キーのみ |
| events | `events/<domain>/part-<uuid>.jsonl` | **既存 JSONL に追記せず新規 part 作成**（schema README §4-6） |
| manifests | `manifests/<artifact>/<run_id>.json` · `manifests/latest/<artifact>` pointer | pointer は方式 B（新 snapshot 生成 + pointer 更新） |
| snapshots | `snapshots/<projection>/<run_id>/set.parquet` | 新規 run_id |

**正本キー = 履歴全体**。`manifests/latest/` は投影（SSOT ではない）。

### 2.3 append-only 規約（不変条件）

1. **INSERT ONLY** — UPDATE / DELETE を表す列・操作を作らない（FOUND-D02）。
2. **訂正・取消も新イベント** — `action` / `event_reason` を持つ新イベントで表現（schema README §4-3）。
3. **状態は集計で** — 「現在のタグ」「現在残高」を固定列にしない。snapshot で再計算（FOUND-N07）。
4. **provenance 必須** — 派生成果物に `run_id` · `schema_version` · `input_hash` · `model_*`（FOUND-D07）。
5. **value_origin で由来区別** — 直接観測 / 画像由来 / 環境由来 / 推定を混同しない（FOUND-D08 · FOUND-N08 前提）。

## 3. API / 契約（ライブラリ公開 API · route 契約）

> #00 は HTTP route を直接持たない（土台ライブラリ）。ここでは**全機能が依存する公開 API 契約**を route 契約に準じて固定する。破壊的変更は CR + schema_version +1。

### 3.1 `SchemaRegistry`（`libs/ihl/core/schema_validator.py`）

| メソッド | 入力 → 出力 | 失敗時 |
|----------|-------------|--------|
| `validate(schema_ref, instance) -> ValidationResult` | `(str, dict)` → `ValidationResult(ok, issues[])` | issues に path/message 集約（例外なし） |
| `validate_or_raise(schema_ref, instance) -> None` | 同上 | `jsonschema.ValidationError`（path:message 連結） |
| `schema_exists(schema_ref) -> bool` | `str` → `bool` | False（不正 ref は ValueError 捕捉） |
| `load_schema_doc(schema_ref) -> dict` | `str` → resolved schema | `FileNotFoundError` / `ValueError`（非 mapping） |
| `dictionary_keys(rel_path) -> list[str]` | `str` → enum keys | `FileNotFoundError` |

- `schema_ref` 正規化: 拡張子なしは `{ref}.schema.yaml` を付与（`_normalize_ref` L140–144）。
- schemas_root 解決順: `IHL_SCHEMAS_ROOT` → `REPO_ROOT/schemas` → `02-設計/_横断/schema/schemas` →（fallback）（`default_schemas_root` L46–68）。

### 3.2 `R2Client`（`libs/ihl/core/r2_io.py`）

| メソッド | 契約 | エラー |
|----------|------|--------|
| `write_bytes(key, data)` | overwrite=False 固定 | `R2NoOverwriteError`（既存キー衝突） |
| `write_json(key, payload)` | UTF-8 · indent=2 · ensure_ascii=False | 同上 |
| `write_parquet_bytes(key, data)` | バイナリ追記専用 | 同上 |
| `read_bytes(key) / read_json(key)` | bytes / dict | `R2NotFoundError`（キー不在） |
| `list_keys(prefix, max_keys=200)` | 新しい順（local）| 空 list（prefix 不在） |

- backend 選択: `IHL_R2_LOCAL_ROOT` 設定 or `r2_configured()==False` → `LocalFilesystemBackend`、それ以外 `Boto3Backend`（`R2Client.__init__` L185–200）。
- `r2_configured()` は placeholder（`<account_id>` · `your_r2_*` · `changeme`）を live と誤認しない。

### 3.3 `catalog`（`libs/ihl/core/catalog.py`）

| 関数 | 契約 |
|------|------|
| `load_components(path=None) -> list[dict]` | `catalog/components.yaml` の `components`（C-USB 参照） |
| `pipeline_order(path=None) -> list[str]` | Makefile / run-pipeline 実行順 |

### 3.4 エラーコード規約（共通）

| 種別 | 例外 | 意味 |
|------|------|------|
| 上書き禁止違反 | `R2NoOverwriteError`（`FileExistsError`） | append-only 破り検知 |
| キー不在 | `R2NotFoundError`（`FileNotFoundError`） | read 対象なし |
| 契約検証失敗 | `jsonschema.ValidationError` | schema 不適合 |
| schema 不在 | `FileNotFoundError` | schema_ref 解決失敗 |

## 4. 状態機械（主要 state · 永続化 · 冪等）

### 4.1 R2 オブジェクトのライフサイクル

```
(absent) --write_bytes(key)--> (persisted, immutable)
(persisted) --write_bytes(same key)--> R2NoOverwriteError（状態遷移しない）
(persisted) --read_bytes(key)--> (persisted)  ※読み取りは状態を変えない
```

- **不変条件**: persisted から persisted への上書き遷移は**存在しない**（append-only）。
- **冪等性**: 同一 `key` の再書込は必ず失敗（衝突検知）。再実行は**新 run_id / 新 part-uuid** で別キーとして実現する。

### 4.2 schema_version の状態遷移（append-only on schema）

```
schema_version: N  --後方互換な任意列追加-->  schema_version: N（据え置き）
schema_version: N  --破壊的変更(列削除/型変更/required追加/enum削除)-->  schema_version: N+1（旧版は残す）
```

- 旧 version の schema / データは消さない。読み手が `schema_version` 別に解釈（再解析可能性 · FOUND-N06）。
- 破壊検知は CI（`02-設計/_横断/ci/CI設計書-v1.md` §schema break）。

### 4.3 latest pointer（方式 B 確定）

```
新 snapshot 生成(run_id) --> manifests/latest/<artifact> pointer を新規作成/更新
```

旧 snapshot は不変のまま残置。pointer は投影であり Truth ではない（ADR-D01）。

## 5. component ITO（IN → Transform → OUT · manifest 参照）

### 5.1 ITO 原則（C-USB）

すべての Component は `Input → Transform → Output` を宣言し、`io.role`（input / transform / output / mixed）と `io.inputs[]` / `io.outputs[]` を持つ（C-USB.md · ComponentFramework §1）。

### 5.2 土台が保証する ITO 基盤

| 段 | #00 の責務 | 契約 |
|----|-----------|------|
| IN | input_manifest 検証 | `manifest/input_manifest.schema.yaml` を `SchemaRegistry.validate` |
| Transform | provenance 付与 | `common/provenance.schema.yaml`（run_id / schema_version / input_hash） |
| OUT | output_manifest + R2 append-only 書込 | `manifest/output_manifest.schema.yaml` + `R2Client.write_*` |

### 5.3 component catalog 連携

`catalog/components.yaml` の各 component は schema を**参照のみ**（`in_schema_ref` / `out_schema_ref`）。pipeline_order（`ingest_normalize → thumbnail_builder → embedding_builder → manifest_builder`）が実行順の正本。
