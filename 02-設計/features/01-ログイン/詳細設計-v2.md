## 2. データ契約（トークン · セッション · 監査イベント）

### 2.1 マジックリンクトークン（ephemeral · 非 R2）

`AuthSessionStore.pending_links`（dev 実装 · プロセス内）:

| フィールド | 型 | 規約 |
|------------|-----|------|
| `email_hash` | str | `hash_actor_id(email.strip().lower())`（SHA256 系 · 平文非保持） |
| `email` | str | dev store の照合用（lower/strip 正規化 · **ephemeral · 非永続化**） |
| `expires_at` | float | `time.time() + 900`（**TTL 15 分** · FR-LOGIN-02 / NFR-LOGIN-02） |

- **ワンタイム**: `verify_magic_link` は `pending_links.pop(token)` で**消費＝即削除**（再利用不可 · NFR-LOGIN-02）。
- **トークン値**: `secrets.token_urlsafe(32)`（暗号論的乱数）。

### 2.2 セッション（ephemeral · 非 JWT）

`AuthSessionStore.sessions`:

| フィールド | 型 | 規約 |
|------------|-----|------|
| `actor_id` | str | `pending["email_hash"]`（ハッシュ済 · 平文 email を持たない） |
| `created_at` | float | 発行時刻 |

- セッショントークンは `secrets.token_urlsafe(32)`。`resolve_session(session_token)` が `actor_id` を返す（不在は `None`）。
- **IHL dev は opaque session_token**（legacy の `localStorage` JWT とは別方式 · §7 retrofit 差分参照）。

### 2.3 監査イベント（append-only · Truth）

`magic-link` 要求時に `write_pii_access_event` を 1 件追記:

| フィールド | 値 |
|------------|-----|
| `actor_id` | `hash_actor_id(email)`（平文 email を残さない） |
| `access_kind` | `read` |
| `target_ref` | `auth_magic_link` |
| `legal_basis` | `user_self` |

- **append-only**（#00 土台規約 · UPDATE/DELETE 禁止）。訂正も新イベント。

## 3. API / route 契約（IHL 実エンドポイント）

> prefix = `/api/v1/auth`（`router = APIRouter(prefix="/api/v1/auth")`）。破壊的変更は CR + バージョン。

### 3.1 `POST /api/v1/auth/magic-link`（FR-LOGIN-01/02/03）

| 項目 | 契約 |
|------|------|
| body | `{ "email": str }`（`MagicLinkRequest`） |
| 副作用 | `write_pii_access_event`（§2.3）· `issue_magic_link`（§2.1） |
| 200 応答 | `{ "status": "sent", "message": "ログインリンクを送信しました（開発モード）" }` |
| dev 拡張 | `IHL_DEV_EXPOSE_MAGIC_TOKEN == "1"` のとき `dev_token` を追加（FR-LOGIN-03） |

- 本番では `dev_token` を返さない（env 未設定時 · NFR-LOGIN-03 可用性は dev/CI のみで担保）。

### 3.2 `POST /api/v1/auth/verify`（FR-LOGIN-04）

| 項目 | 契約 |
|------|------|
| body | `{ "token": str }`（`MagicLinkVerifyRequest` · `token.strip()`） |
| 成功 200 | `{ "status": "authenticated", "session_token": str, "actor_id": str }` |
| 失敗 401 | `detail = "INVALID_OR_EXPIRED_TOKEN"`（不正 · 期限切れ · 使用済みを集約） |

- 検証成功でトークンは消費済（再 verify は 401）。

### 3.3 `POST /api/v1/auth/register`（FR-LOGIN-01 同意境界 · #03 連携）

| 項目 | 契約 |
|------|------|
| body | `{ "handle": str, "language": str = "ja", "agree_terms": bool = false }` |
| 同意なし 400 | `detail = "利用規約への同意が必要です"`（`agree_terms == false`） |
| 成功 200 | `{ "status": "registered", "handle": str }` |

### 3.4 エラーコード規約（共通）

| 種別 | HTTP | detail | 要件 |
|------|------|--------|------|
| トークン不正/期限切れ/使用済み | 401 | `INVALID_OR_EXPIRED_TOKEN` | FR-LOGIN-04 |
| 規約未同意 | 400 | 利用規約への同意が必要です | FR-LOGIN-01 |
| （legacy 参照）レート超過 | 429 | `RATE_LIMIT` | FR-LOGIN-09（IHL 未実装 · §7 ギャップ） |

## 4. 状態機械（magic link → session ライフサイクル）

### 4.1 トークンのライフサイクル

```
(absent) --issue_magic_link(email)--> (pending, expires_at=now+900s)
(pending) --verify within TTL-------> (consumed) + session 発行
(pending) --verify after TTL--------> None（401）· pending は pop 済で破棄
(pending) --verify wrong token------> None（401）
(consumed) --verify same token------> None（401 · 再利用不可）
```

- **不変条件**: 1 トークン = 1 回の verify のみ成功（`pop` による消費 · NFR-LOGIN-02）。
- **冪等性**: 同一トークンの再 verify は必ず失敗。再ログインは**新規 magic-link 発行**で行う。

### 4.2 セッションのライフサイクル

```
(no session) --verify success--> (session_token 発行, actor_id 紐付け)
(session)     --resolve_session--> actor_id（読み取りは状態を変えない）
(session)     --store reset(tests)--> (no session)
```

- dev store はプロセス内・`reset_auth_session_store()` でテスト間クリア（`tests/unit/test_auth.py`）。

### 4.3 クライアント遷移（[`遷移設計-v1.md`](./遷移設計-v1.md) と整合）

```
[idle] --email+terms--> [sending] --200--> [sent]（dev_token あれば「このトークンで認証」）
[sent] --verify OK--> [authenticated] --> #03 onboarding or #04 home
[*]    --verify fail--> [verify_error] --> idle（再送信導線）
```
