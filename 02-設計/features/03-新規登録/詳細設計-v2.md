## 0. この文書の位置づけ

本書は **要件 #03（新規登録）の `FR-REG-*` / `NFR-REG-*` を「実装と検証で追える設計」に厚くした v2** である。

- **重要な読み替え（IHL rebuild）**: 要件 #03 は legacy civilization-os（`OnboardingFlow.tsx` · `authLogic.setupProfile` · R2 `users/{userId}.json` · `localStorage` JWT）を母体に書かれている。IHL では「stays in civilization-os」を **IHL rebuild（legacy = salvage 参照）** と読む。**現行の実装正本は Python FastAPI**（`apps/api/routes/onboarding.py` の `onboarding_complete` / `onboarding_status` + `auth.py` の `auth_register`）であり、legacy TS は **設計意図の salvage 参照**として扱う。
- **新規登録の本質**: 独立サインアップ画面はない。#01 の magic-link 検証でアクター成立 → **初回 handle 確定（オンボーディング）** で「文明内の人格」が成立する。IHL dev では handle 確定を `onboarding_complete` イベントで表現する。

## 2. データ契約（オンボーディング完了 · 登録）

### 2.1 オンボーディング完了記録（`onboarding_completed_v1` · 永続）

`onboarding_complete` は完了レコードを 1 件保存:

| フィールド | 型 | 規約 |
|------------|-----|------|
| `schema` | str | `onboarding_completed_v1` |
| `handle` | str | `strip()` 済 · **最低 2 文字**（FR-REG-07 簡略 · IHL dev） |
| `language` | str | 既定 `ja`（locale 必須 · FR-REG-06a / #21） |
| `actor_id` | str | 既定 `u_demo`（本番 hash 前提 · PII 最小化） |

- 保存先: `default_event_root() / "onboarding" / "v1" / {actor_id} / "completed.json"`。
- **冪等性**: 既存 `completed.json` があれば 409 `ONBOARDING_ALREADY_COMPLETE`（再完了禁止 · §4）。

### 2.2 登録境界（`auth_register` · #01 連携）

| フィールド | 型 | 規約 |
|------------|-----|------|
| `handle` | str | 登録 handle |
| `language` | str | 既定 `ja` |
| `agree_terms` | bool | **`true` 必須**（false で 400） |

- 登録は #01 の `RegisterRequest` を共有。同意なしは 400「利用規約への同意が必要です」。

## 3. API / route 契約（IHL 実エンドポイント）

> onboarding prefix = `/api/v1/onboarding`。register は #01（`/api/v1/auth/register`）。

### 3.1 `POST /api/v1/onboarding/complete`（FR-REG-12/13/15）

| 項目 | 契約 |
|------|------|
| body | `{ "handle": str, "language": str = "ja", "actor_id": str = "u_demo" }`（`OnboardingCompleteBody`） |
| handle 不正 400 | `detail = "HANDLE_REQUIRED"`（空 or 2 文字未満） |
| 二重完了 409 | `detail = "ONBOARDING_ALREADY_COMPLETE"` |
| 成功 201 | `{ "status": "complete", "handle": str }`（`completed.json` 追記） |

### 3.2 `GET /api/v1/onboarding/status`（FR-REG-05/22）

| 項目 | 契約 |
|------|------|
| query | `actor_id: str = "u_demo"` |
| 未完了 200 | `{ "status": "pending", "complete": false }` |
| 完了済 200 | `{ "status": "complete", "complete": true, "handle": str }` |

### 3.3 `POST /api/v1/auth/register`（FR-REG-17/19/21 · #01 境界）

| 項目 | 契約 |
|------|------|
| body | `{ "handle": str, "language": str = "ja", "agree_terms": bool = false }` |
| 同意なし 400 | `detail = "利用規約への同意が必要です"` |
| 成功 200 | `{ "status": "registered", "handle": str }` |

### 3.4 エラーコード規約（共通）

| 種別 | HTTP | detail | 要件 |
|------|------|--------|------|
| handle 不正 | 400 | `HANDLE_REQUIRED` | FR-REG-07 |
| 二重完了 | 409 | `ONBOARDING_ALREADY_COMPLETE` | FR-REG-13 |
| 規約未同意 | 400 | 利用規約への同意が必要です | FR-REG-19/21 |
| （legacy）handle 重複 | 400 | `HANDLE_TAKEN` | FR-REG-13（IHL 未実装 · §7 P4） |
| （legacy）handle 変更 | 400 | `HANDLE_IMMUTABLE` | FR-REG-14（IHL 未実装 · §7 P4） |

## 4. 状態機械（オンボーディング ライフサイクル）

### 4.1 完了のライフサイクル

```
(pending: status=pending complete=false)
   --POST /onboarding/complete(handle≥2)--> (complete: completed.json 生成)
(complete) --GET /status---------------> { complete: true, handle }
(complete) --POST /onboarding/complete--> 409 ONBOARDING_ALREADY_COMPLETE（冪等 · 上書き禁止）
(pending)  --POST /complete(handle<2)---> 400 HANDLE_REQUIRED（状態不変）
```

- **不変条件**: 1 アクター = 1 回の完了のみ（`completed.json` 既存判定 · NFR-REG-04 二重送信防止のサーバ側担保）。
- **append-only**: 完了レコードは追記（訂正は別キー · ProjectRules INSERT ONLY）。

### 4.2 クライアント遷移（[`遷移設計-v1.md`](./遷移設計-v1.md) と整合）

```
[verify OK(#01)] --status pending--> [onboarding]（handle + language + theme 入力）
[onboarding] --complete 201--> [home(#04)]
[onboarding] --handle 不正--> [onboarding(error)]（理由表示 · 再入力）
[onboarding] --別メール--> ログアウト → /login（#01）
```

## 5. component ITO（IN → Transform → OUT）

| 段 | #03 の責務 | 契約 |
|----|-----------|------|
| IN | handle + language + actor_id（complete）/ actor_id（status）/ handle+agree（register） | pydantic `OnboardingCompleteBody` / `RegisterRequest` |
| Transform | handle 検証 · 冪等判定 · 完了レコード追記 · 規約同意検証 | `event_store` + `default_event_root` |
| OUT | `complete` + handle / `pending` / `registered` | JSON 応答（§3） |

- C-USB 上の位置づけ: FeatureNode `auth`（Kernel `create`=初回行 · `configure`=locale/theme）。IHL では route + event_store として実装（要件 §⑥ 整合）。
