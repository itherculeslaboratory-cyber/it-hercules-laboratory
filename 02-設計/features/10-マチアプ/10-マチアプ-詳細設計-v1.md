## 2. preference_event JSON schema（正本候補 — `02-設計/_横断/schema/preference_event.yaml`）

append-only JSONL（R2 `events/preference_event.jsonl` — ADR-H-03）。

```json
{
  "preference_event_id": "pe_01J...",          // pe_{ulid}
  "schema_version": "preference_event_v1",
  "kind": "pairwise_choice",                    // pairwise_choice | binary_image | dimension_matrix
  "user_id": "u_...",                           // JWT 主体
  "session_id": "ps_...",                        // 好みセッション（任意）
  "created_at": "2026-06-07T10:00:00Z",         // ISO8601

  // --- kind = pairwise_choice ---
  "left_asset_id": "cap_...",                   // capture_id 等
  "right_asset_id": "cap_...",
  "choice": "left",                              // left | right | neither | skip

  // --- kind = dimension_matrix（ValueCheck 詳細のみ） ---
  "template_id": "valuecheck_default_v1",
  "dimensions": [
    { "key": "horn", "value": "circle" },        // x | minus | circle | null
    { "key": "color", "value": "x" }
  ],
  "overall_fit": "yes",                          // yes | maybe | no（dimension_matrix 時）

  "source": "match_ui",                          // match_ui | valuecheck_detail
  "context": {
    "pair_strategy": "uncertainty",              // ペア選択アルゴリズム識別（§5）
    "capture_set_snapshot_id": "snap_..."        // 提示母集団の snapshot
  }
}
```

### 2.1 kind 別必須フィールド

| kind | 必須 | source |
|------|------|--------|
| `pairwise_choice` | left_asset_id, right_asset_id, choice | `match_ui` |
| `binary_image` | left_asset_id（対象）, choice ∈ {left=YES, neither=NO} | `match_ui`（legacy 移行のみ） |
| `dimension_matrix` | template_id, dimensions, overall_fit | `valuecheck_detail` |

> **不変条件 INV-PE-01**: append-only（修正は新イベント・invert 相当を追加、上書き禁止 — F4/F9）。
> **INV-PE-02**: `binary_image` は移行期のみ。新規記録は `pairwise_choice` 優先（ADR-H-02）。
> **INV-PE-03**: `dimension_matrix` の writer は **ValueCheck 詳細画面のみ**（本線 pairwise は MatchApp 系）。

## 3. capture_id 紐づけ

| 項目 | 設計 |
|------|------|
| `left_asset_id` / `right_asset_id` | IHL `capture_id`（観測 searchable_capture_set の主キー）を指す |
| 解決元 | `manifests/latest/searchable_capture_set`（05 観測）から候補 capture を取得 |
| civ-os 観測との関係 | sessionId ↔ capture_id の ID マッピングは **H-06（未決）**。Phase 1 は IHL capture のみ対象 |
| snapshot 固定 | `context.capture_set_snapshot_id` で提示時の母集団を記録（再現性・F8 provenance） |

> capture 接続の確定は H-06（個体 ID 移行）と連動。Phase 1 は IHL 内 capture に閉じる。

## 4. aggregate / rerank パイプライン（スケッチ）

`components/preference_aggregator/run.py`（[フォルダ構成 ADR](./ADR-Phase1-IHL-repoフォルダ構成.md)）。

```text
[1] read   events/preference_event.jsonl（user_id でフィルタ・直近 N）
[2] derive  pairwise → ペアワイズ勝敗をタグ/次元重みへ写像
            - choice=left → left の特徴 +1 / right -1（相対）
            - neither     → 両者 -（弱い負）
            - dimension_matrix → ValueCheck と同じ avoid/prefer 写像
[3] merge   ValueCheck（dimension_matrix）と pairwise を統合
            - 衝突時 avoid 優先（FR-MCH-07 と同規則）
[4] output  preference_profile（user 単位・snapshot）
            → 検索 rerank の boost コンテキスト（searchable rerank へ供給）
[5] manifest run_info（run_id/model/input_hash）+ errors.jsonl
```

| 出力 | 内容 |
|------|------|
| `preference_profile`（Parquet/JSON snapshot） | user_id · 推定 prefer/avoid 特徴 · 収束度 · 寄与イベント数 |
| rerank 反映 | 05 観測 searchable rerank の boost 項（重みは **H-05 待ち** — D-02 と整合） |

> **集約は append-only ビュー再生成**（上書き修正禁止 — F9 / FR-MCH-16）。過去イベントは消さない。

## 5. pairwise ペア選択アルゴリズム（note）

2 枚をどう選ぶかで学習効率が変わる。Phase 1 は軽量、Phase 2 で高度化。

| 戦略 | 内容 | Phase |
|------|------|-------|
| `random` | 母集団から一様 2 枚 | Phase 1 既定（cold start） |
| `uncertainty` | 現 profile で **接戦**（推定価値が近い）2 枚を選び情報量を最大化 | Phase 1 後半 |
| `diversity` | 既提示と特徴が離れたペアで探索 | Phase 2 |
| `active`（FAISS） | embedding 近傍から境界ペア抽出 | Phase 2（FAISS 導入後） |

| ルール | 内容 |
|--------|------|
| 重複回避 | 同一 (left,right) ペアの短期再提示を避ける（session 内） |
| 収束判定 | 直近 K 回の profile 変化量が閾値未満 → 「収束」状態（遷移設計 §convergence） |
| 母集団 | `context.capture_set_snapshot_id` で固定し再現可能に |
| `pair_strategy` 記録 | 各 event に採用戦略を残す（後分析・F8 provenance） |

## 6. API エンドポイント（スケッチ — 既存契約に整合・新 invent は実装フェーズ）

| Method · Path | 用途 | 認証 | 備考 |
|---------------|------|:----:|------|
| `GET /api/match/next-pair` | 次の 2 枚（pair_strategy 適用） | 要 | response: left/right capture + pair token |
| `POST /api/match/choice` | pairwise 選択を記録 | 要 | body: left/right/choice → `preference_event` append |
| `GET /api/match/profile` | 自分の preference_profile（収束度・topTags） | 要 | aggregator 出力の read |
| `GET /api/value-check/*` | ValueCheck テンプレ/セッション（既存 FR-MCH-04） | 要 | `dimension_matrix` writer |
| `POST /api/value-check/session` | ValueCheck 詳細記録 | 要 | `preference_event(dimension_matrix)` + `ValueCheckSessionRecord` |

> **境界**: `match_ui` 系は pairwise（本線）、`valuecheck_detail` 系は dimension_matrix（詳細のみ）。両者とも単一 `preference_event` ストリームへ（ADR-H-02）。
