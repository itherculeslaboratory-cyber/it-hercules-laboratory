## 1. レビュー目的

U-MKT-DSP-01〜04 の **詳細設計たたき台** が、IHL 全システムと **衝突なく整合** しているかを検証する。

**レビュー後の期待成果**

- 各チェック項目に **PASS / FAIL** と **修正提案**（FAIL 時）
- FAIL が設計ゲート阻害レベルなら **本書または詳細設計 v1 への具体的 diff 指示**
- 人間レビュアが **短時間で Go/No-Go** できる要約（1 ページ以内）

## 3. ユーザー確定で動かせない前提（固定）

レビューで **変更提案してはいけない** 確定事項。矛盾を見つけたら **詳細設計側を直す** のではなく **FAIL + 「§14 違反」** と報告すること。

| # | 固定前提 | 出典 |
|---|----------|------|
| P1 | **開発者・運営は裁判官ではない** — 判定権限 UI なし | §14.1 · NFR-DSP-01 |
| P2 | **非エスクロー** — OS は返金・配送代行しない | §14.7.1 · `mkg_p2p_core` |
| P3 | **指摘 = タグ + 理由必須** · 通報ボタンなし | §3.2 · §14.2 |

### 4.1 衝突検出

| ID | 観点 |
|----|------|
| C-01 | `dispute-room` 統合が **07 掲示板 §3** と矛盾しないか |
| C-02 | `trade_private` 非公開が **06 §11.0 Stage 1** と一致するか |
| C-03 | `fee_unpaid` / GMO 8% が **争い API・判例** と誤接続されていないか |
| C-04 | **PT 投票** と **指摘 30 回/PT** が **22 単一台帳** と整合するか |
| C-05 | **取引成立**（§11.0.1）と Y09「支払後」タイミングの **用語** が詳細設計内で混同されていないか |
| C-06 | **Y11 行政** が争いフローに **誤って含まれていないか** |

### 4.2 漏れ

| ID | 観点 |
|----|------|
| G-01 | **ネスト指摘**（深さ無制限）が anchor / parent_dispute_id で表現されているか |
| G-02 | **1 錨 1 open** が board / listing / trade / rating すべてに適用されているか |
| G-03 | **公開エクスチェンジ → 投票 → 判例** のイベント連鎖が R2 schema にあるか |
| G-04 | **評価版管理**（Y08）が `content_revision_applied` と整合するか |
| G-05 | **証拠 attach**（§6.8）が `dispute-evidence-attach` と接続されているか |
| G-06 | **翻訳**（21）— 部屋メッセージのクライアント翻訳が API 設計を壊さないか |

### 4.3 過剰複雑

| ID | 観点 |
|----|------|
| X-01 | **15 タグ** に戻る余地がなく、7+trade_subtag で **Y01〜Y15 をカバー** できるか |
| X-02 | **別 dispute コンポーネント** を増やす提案が紛れ込んでいないか |
| X-03 | **trade 錨の msg 粒度** — v1 trade 単位で十分か、過剰/不足か |
| X-04 | API エンドポイント数は **最小** か — 統合可能なものが分裂していないか |

### 4.4 カルマ二層整合

| ID | 観点 |
|----|------|
| K-01 | `delta_count_hint` が **開始時自動適用** になっていないか（確定イベントのみか） |
| K-02 | 掲示板 **未解決 5 倍数 +1** とマーケット **即時 +1/+3/+5** の **累計軸分離** が明確か |
| K-03 | 投票敗者 +5 が **二重付与** されうる経路がないか |
| K-04 | Y02 期限切れ（争い室なし）と **trade_private 争い** の Δcount が重複しないか |
