## 7. 「大幅変更」前に押さえる論点

1. **スコープの芯**: OS 内 **統治争いの記録・審理**（外国裁判非代替）— 利用規約 §14 と一致させるか。
2. **カルマとの境界**（最重要）: 確定設計は **司法で karma を動かさない**。変更するなら `karma-judicial-correspondence`・middleware・FAQ 一式の再設計が必須。
3. **停止の意味**: 「アカウント停止」は実際には **ミューテーション API 403** + 回復導線— 法的表現と技術の整合。
4. **審理者モデル**: env 固定 ID vs コミュニティ審理 vs 「創世は裁判官にならない」— どれを正とするか。
5. **掲示板＝司法** の遺産: `/board/justice`、DM 統合、公開スレッド— API 中心 UI に寄せた結果とのギャップ。
6. **信頼（judicialTrust）の目的**: カルマ・貢献度・市場評価との **役割分担**（CH06「即席スコア却下」意図）。
7. **自動段階進行（cron 72h）**: 無操作で審理が進む設計思想— 廃止・人手化・コミュニティ化の選択。
8. **案件タイプ拡張**: `trade_dispute` 等を足すか、general に吸収するか（fb_00246 の未回収）。
9. **IHL 境界（2026-06-07 確定）**: **単一 IHL repo** が唯一の新実装先。legacy civ-os の judicial API/UI は **salvage 参照のみ**；裁判は **IHL repo 内 component として rebuild**（`05-運用/_横断/リポジトリ戦略-legacyとIHL.md`）。二製品並行・consumer 接続・IHL 内 duplicate 禁止（旧 stays モデル）は **廃止**。
10. **Builder/L4**: `/admin/judicial` は L4 寄り— Builder から黙示的に変えない（`builder-capability-boundary.md`）。IHL rebuild 時は L4 概念自体を再検討。
11. **通知・インボックス**: REQ-008 と一体— 司法変更は通知チャネル契約に波及。
12. **テスト・出荷ゲート**: REQ-005 implemented 宣言、E2E は浅い（INV-028）— 変更後の証跡戦略。
