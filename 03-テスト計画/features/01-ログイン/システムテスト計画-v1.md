## 1. 目的・範囲

ログインが**システム全体として一貫**しているか（メール入力 → トークン取得 → 検証 → 認証済 → 次フロー）を担保する。システム層は **(a) ログイン E2E フロー** と **(b) 設計↔実装 parity / PII 静的整合** の 2 系統で構成する。

## 4. カバレッジ方針

- E2E（ST-01-01/02/03）で **ログイン主経路 + ワンタイム + 次フロー接続**（FR-LOGIN-04/07 · NFR-LOGIN-02）。
- 静的検査（ST-01-04/05/LINT-06）で **parity · PII 最小化 · 構造規範**を担保。
- 規範系 TC は手動レビュー + grep を併用し、結果を RTM `status` に記録。
