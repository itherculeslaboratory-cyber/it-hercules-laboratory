## 1. 目的・範囲

ログイン route（`/api/v1/auth/*`）が **session store · PII · event store と整合**して動くことを `fastapi.testclient.TestClient` で担保する。1 関数内部（単体）と E2E（システム）の中間層。

| 範囲 | 対象 |
|------|------|
| In | magic-link → verify → register の HTTP 経路 · dev_token 露出 · 監査イベント追記 · エラーステータス |
| Out | 関数内部分岐（単体）· ブラウザ UI フロー（システム） |

---

## 2. テスト対象と既存資産（retrofit）

| 対象 | 既存テスト（正本） |
|------|--------------------|
| magic-link/verify route | `tests/unit/test_auth.py::test_magic_link_and_verify` |
| register 規約同意 route | `tests/unit/test_auth.py::test_register_requires_terms` |
| event store 追記 | `tests/unit/test_event_store.py` · `tests/integration/test_api_events.py` |

> 注: 既存 `test_auth.py` は `unit/` 配下だが TestClient を用いる **route 結合**性質を持つ。本計画では結合 TC として参照し、純内部ロジックは単体計画 UT に分離した。

## 4. カバレッジ方針

- route × store の **HTTP 契約 + 副作用**（FR-LOGIN-01/03/04 · NFR-LOGIN-01/06）を中心に担保。
- ステータスコードだけでなく **応答 body / event 内容**を assert（C3 準拠 · 200 OK のみで PASS としない）。
- FR-LOGIN-09（レート制限）は IHL 未実装（v2 §7 P4）のため、本層では**設計ギャップ**として RTM `status=gap`。将来 retrofit で IT 追加。
