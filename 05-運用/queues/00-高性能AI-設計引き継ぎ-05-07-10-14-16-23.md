## 1. 目的とスコープ

### 1.1 目的

7 機能の **要件定義たたき台** を、**設計ゲート 4 点**（要件 / 詳細 / 遷移 / UI）が人間確定できる粒度へ引き上げる。**実装は禁止**（`.cursor/rules/design-before-implementation-gate.mdc`）。本セッションでの判定: **要件層は書ききれている（YES）が、設計ゲートは全機能 PARTIAL**（詳細/遷移/UI が空白）。

### 1.2 スコープ

| する | しない |
|------|--------|
| 要件の矛盾解消・確定／詳細設計（schema・API・契約）／遷移設計（state machine）／UI 設計（ワイヤー）のドキュメント化 | フロント/バック/IHL repo の**実装コード**変更 |

## 4. 各機能で次に作るもの（詳細 / 遷移 / UI）

### 05 観測
- **詳細**: 2026.06,06 schema を YAML 化（`searchable_capture_set` 全列・`environment_timeseries`・`value_origin` 辞書）。civ-os 固体 commit ↔ IHL `capture_id` の ID マッピング ADR。
- **遷移**: run → snapshot → latest 採用の append-only 状態（旧 run 残存）。
- **UI**: IHL Streamlit（filter sidebar → 結果 grid → 詳細 pane → 類似検索 pane → タグ pane）。civ-os 固体フローは別ワイヤー。**色補正しない・撮影条件併記**（`preferences.md` §C）。

### 06 マーケット
- **詳細**: `trade-events` / `listing-state` イベント列定義。8% 期待入金は 23 を参照。
- **遷移**: Listing 状態機械（許可辺表）+ Auction(open→settled) + Stage 0〜3 + 評価ウィンドウ(最大1ヶ月→自動「良い」)。
- **UI**: 出品/オークション/Engagement（公開）+ プライベートボード（Stage1 非公開）+ 公開エクスチェンジ + 評価ボード。

### 07 掲示板
- **詳細**: スレ/投稿/指摘イベントのデータ契約。`dispute_kind=board_pointer` 等は 11 を参照。
- **遷移**: 指摘→二人部屋→クローズ（11 §3 と整合・1 錨 1 open）。
- **UI**: 製品 BBS（愚痴/改善/論文/その他）+ 投稿 rescue（失敗理由・再試行・保存境界）。国際 BBS は 21 クライアント翻訳。

### 10 マチアプ
- **詳細**: ValueCheck は実装参照で確定。**MatchApp↔capture データ契約**は ADR 後に確定。
- **遷移**: MatchApp 評価ループ（YES/NO→重み更新→収束）の状態。
- **UI**: 評価/ランキング/タグ/分析の 4 画面。緑/赤ボタンは fb_00046 理想（色は意味のみ）。**stub 明示**。

### 14 貢献度
- **詳細**: `NodeGraphEntry`・`economy/graph/` schema。upstream%/100単位パラメータの IHL 確定。
- **遷移**: N/A（イベント換算）。
- **UI**: ホーム要約・コンポーネント詳細バッジ（換算式は短い説明+ヘルプ）。

### 16 UIbuilder
- **詳細**: §④ vs FR-16-REFRAME の**採用判断後**に catalog 紐づけ契約。fork v2 entity_kind は backend 正本維持。
- **遷移**: Builder→catalog 選択→preview→（任意）fork Apply。機能開発は /dev 外導線。
- **UI**: 配置・デザイン中心の IDE ワイヤー。RAG/fork/Pilot タブの去留を決める。

### 23 GMO
- **詳細**: `deriveTransferCode` を schema/関数契約化。`ExpectedPaymentRecord` の OpenAPI/schema。照合アルゴリズム（§2.5.3）の確定。
- **遷移**: expected_payment(pending→matched / partial→継続)。
- **UI**: 取引成立後の振込案内プロンプト（振込先・金額・コード・コピー可）。

## 5. 変更してはいけない（確定済み · 動かさない）

レビュー中にこれらと矛盾を見つけたら、**確定側を直すのではなく FAIL として報告**する（11 §3 固定前提 P1〜P12 と整合）。

| # | 確定事項 | 正本 |
|---|----------|------|
| C1 | **取引成立** = 配送完了確認 **かつ** 評価確定（1ヶ月無評価で自動「良い」）。マッチング/`settled`/`sold` 単独では成立しない | 06 §11.0.1 · FR-MKT-13 |
| C2 | **8% は取引成立後のみ起算** · 売り手義務 · 月次 Fibonacci Δcount · 争い非経由 | 06 §11.7 · 08 §5.1 |
| C3 | **振込コード = `deriveTransferCode(userId)`**（SHA-256→uint24→Base36→`U-` pad/truncate）· ユーザーごと 1 コード（8%/PT 共通） | 23 §2.2 · §2.4 |
| C4 | **複数 pending 衝突 = 入金日時 + `created_at` FIFO**（取引サフィックスは却下・例外フォールバックのみ） | 23 §2.5（U-GMO-06 解決） |
| C5 | **部分入金=残債減算 / 過入金=貢献費クレジット / 返金不可** | 06 §11.7.5 · 23 §3.1 |
| C6 | **Engagement 争いは掲示板指摘と同型** — 取引専用フローを設けない | 07 FR-BBS-11 · 11 §6.5（U-MKT-DSP-09） |
