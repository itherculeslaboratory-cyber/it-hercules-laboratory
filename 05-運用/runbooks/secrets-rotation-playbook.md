## 0. 共通原則

| 原則 | 内容 |
|------|------|
| 載せない | 秘密値は R2・フロント・ログに **絶対に載せない**（平文ログ禁止・FR-SET-09）|
| 管理場所 | サーバ env / GitHub Actions Secrets / collector `.env`（ローカル/Docker volume）|
| ローテ手順 | **新鍵を追加 → 両対応期間（grace）→ 旧鍵を失効** の 3 段（無停止）|
| 記録 | ローテ実施は変更管理に記録（日時・実施者・対象・影響確認）。鍵本体は記録しない |
| 緊急 | 漏洩疑いは grace を省略し **即時失効 → 再発行**（影響を受ける利用者へ再登録案内）|

## 1. JWT_SECRET（認証トークン署名）

**影響**: 既存ログインセッション（JWT）が全て無効化されうる。

```text
[手順]
1. 新 JWT_SECRET を生成（高エントロピー）
2. 検証側を「新+旧の両方で検証可」に（kid/複数鍵対応）→ 署名は新鍵
3. grace（例: アクセストークン有効期限 + α）経過
4. 旧鍵を検証から外す
[確認]
- 新規ログイン → 新鍵で発行・検証 OK
- grace 中の既存トークン → 失効までは通る
[ロールバック]
- 旧鍵検証を残したまま署名鍵だけ旧に戻す
```

> 単一鍵運用しかできない場合: 切替時に **全ユーザー再ログイン**が発生する旨を事前周知。

## 2. VAPID キー（Web Push）

**影響**: 既存 Push 購読が無効化 → 端末の **再登録（再 subscribe）**が必要。

```text
[手順]
1. 新 VAPID 鍵ペアを生成
2. 公開鍵をフロント配布 → 新規購読は新鍵
3. 旧鍵で送信できなくなった購読は「通知が来ない」→ 再登録導線を表示（FR-SET-07 代替導線）
4. grace 後、旧鍵を撤去
[確認]
- 新規購読 → 通知到達
- 旧購読 → 失敗を検知し再登録を促す（画面内通知へフォールバック）
```

## 3. GMO API Client Secret（経済・最重要 / 人間ゲート）

**影響**: 入金照合・8% 振込判定の本番接続。**P0-NEXT-GMO-LIVE-EXEC と同じ Tier D 人間ゲート**。

```text
[手順 — 人間のみ]
1. GMO ポータルで新シークレット発行
2. サーバ env / Actions Secrets を新値に更新（旧値は即時または grace 後失効）
3. Webhook 署名検証（NFR-GMO-01）が新シークレットで通ることを検証
4. テスト入金 or サンドボックスで照合 1 件を確認
[確認]
- 署名不一致は 401・秘密ログ禁止が維持されている
- deriveTransferCode の照合が継続して機能
[注意]
- 会計に関わるため、切替時間帯と pending 照合の扱いを事前計画（合算/部分入金の係属を確認）
```

## 4. SwitchBot TOKEN / SECRET（collector）

**影響**: 環境テレメトリ収集のみ（R2 には鍵を載せない）。

```text
[手順]
1. SwitchBot アプリで新トークン/シークレット取得
2. collector の .env を更新（R2・フロントには絶対に置かない）
3. collector 再起動 → 署名付き ingest が継続することを確認（Ed25519 はローテと独立）
[確認]
- /api/env/collector/ingest が 200・署名 OK
- env-samples に token/secret が含まれていない（FR-ENV-08）
```

> collector の **Ed25519 署名鍵**を変える場合は、サーバ側の公開鍵 allowlist を「新+旧」両対応 → grace → 旧削除。

---

## 5. R2 アクセスキー（本番ストレージ / 人間ゲート）

```text
[手順 — 人間のみ]
1. Cloudflare で新 R2 トークン発行（最小権限スコープ）
2. サーバ/CI env を新値に → put/list/get を検証（同キー再 put が 409 = no-overwrite 維持）
3. 旧トークン失効
[確認]
- append-only / immutability テストが通る（D-01 pointer 解決も）
- CI の本番 embedding 限定実行（手動 dispatch）が新鍵で動く
```

## 6. Tier D チェックリスト（人間サインオフ）

```
[ ] 対象シークレットと影響範囲を特定した
[ ] 新鍵を発行し、両対応(grace)を設定した（無停止 or 周知済み）
[ ] 動作確認（新鍵で発行/検証/照合/ingest が成功）
[ ] 旧鍵を失効した（grace 経過後）
[ ] 秘密値をログ・R2・フロント・コミットに残していない
[ ] 変更管理に実施記録（鍵本体は記録しない）
[ ] 緊急時は grace 省略で即時失効 → 影響利用者へ再登録案内
```
