## 1. スマートフォンブラウザ vs Docker component performance split

### 1.1 現在の設計で確認できた事実

| レイヤー | 定義済み内容 | 出典 |

### 1.3 現在の暗黙設計（モデル読み取り）

```
[Streamlit App（Phase1）]
        ↓ libs/ 経由
[DuckDB + Polars]  ←──  R2 Parquet manifest（latest/）
        ↓
[numpy cosine]  ←──  R2 embeddings.npy（locator 経由）

[Docker Components（バッチ）]  ──→  R2 append-only  ──→ 上記の manifest を更新
```

**問題点**: Phase1 は Streamlit が Python プロセス内で DuckDB を実行するため、スマートフォンからのアクセスはサーバー上の Streamlit アプリを経由する。この場合 Streamlit サーバーのメモリが全クライアントで共有される。データ量が増えた時の挙動が未定義。

### 2.3 設計推奨（ADR 新設要）

以下のパターンのいずれかを ADR で選択する必要がある：

**選択肢 A: Makefile-first（Phase1 継続）**

```
Web UI → （直接ディスパッチしない）
Makefile → Docker component → R2
Web UI → R2 polling for latest manifest
```

- Phase1 の延長。Phase2 でも search のみなら十分。
- 新規パイプラインの起動は手動（CLI）のみ。

**選択肢 B: FastAPI ジョブキュー**

```
Web UI → POST /api/jobs { component: "ingest", params: {...} }
FastAPI → run_id 発行 → Docker exec / subprocess
Web UI → GET /api/jobs/{run_id}/status
```

- Phase2 の自然な形。ただし run_info.json が status SSoT となるか要確認。

**選択肢 C: R2 event-driven（append-only 整合）**

```
Web UI → R2 に job_request.jsonl append
Watch process（バッチ or server-less） → R2 polling → Docker component 起動
```

- R2 append-only 思想と最も整合。ただし低レイテンシ非要求（FOUND-N06）前提。

> **班 B 推奨**: Phase1 は **選択肢 A** で十分。Phase2 移行 ADR に **選択肢 B または C** を記載。**B-06〜B-10 は ADR-Phase2-runtime-dispatch として新設が必要。**

  - スマートフォンブラウザのレイテンシ予算（P95 目標値）
